# Кейс: ревью ветки кастомных колонок каталога

AI-ревью ветки `feature/custom_columns` в gateway перед merge. Задача — расширить VOD-каталог новыми полями (теги, домены, плеер, субтитры, главы, live recording), чтобы фронтенд мог показывать и сортировать кастомные колонки. Ревью выявило 5 Important и 6 Minor замечаний при 0 Critical; основные риски — производительность SQL-запросов и несвязанные изменения в ветке.

Дата: 25 марта 2026.

## Ситуация

Задача Kaiten #56769756 «Настройка колонок в каталоге и трансляциях»: пользователь не может настроить вид каталога — все колонки фиксированы. Бэкенд-часть: расширить ответ VOD-каталога новыми полями, добавить сортировку по ним.

Ветка `feature/custom_columns` в репозитории `integros/gateway`. Diff к `origin/master`: 8 файлов, +94/−59 строк. Основной scope — пакет `workspace/catalog` (4 файла из 8).

## Что нашли

### Important (5)

**1. LATERAL JOIN до пагинации.**
Два `LEFT JOIN LATERAL` (агрегация тегов и доменов через `jsonb_agg`) выполняются внутри подзапроса — до `OFFSET/LIMIT`. PostgreSQL вычисляет агрегацию для всех строк каталога, а потом отбрасывает большинство. При тысячах записей это замедлит запрос.

Рекомендация: перенести LATERAL JOIN наружу (после пагинации) или проверить `EXPLAIN ANALYZE` на реальных данных.

**2. Лишние JOIN в `vodCatalogTotalQuery`.**
Запрос считает `COUNT(*)` и `SUM(size)`, но получил статические JOIN к `workspace.meta` и `workspace.privacy`. Раньше JOIN-ы добавлялись динамически только при наличии фильтров по тегам/плееру/доменам. Теперь они выполняются всегда, даже когда не нужны.

Рекомендация: вернуть условные JOIN для total-запроса или оставить статические только при наличии фильтров.

**3. Потенциальная SQL-инъекция через `SortFieldsMap`.**
Значение `live_recording` в `SortFieldsMap` — SQL-выражение `CASE WHEN meta #>> '{vod, event_id}' IS NOT NULL THEN 1 ELSE 0 END`. Само по себе безопасно (захардкожено), но в `getCatalogOrderQuery` направление сортировки `o.Dir` подставляется через `fmt.Sprintf` без валидации. Если `helpers.OrderSlice` не гарантирует whitelist `asc`/`desc` — это инъекция.

Рекомендация: проверить валидацию `o.Dir` на уровне парсинга `OrderSlice`.

**4. Несвязанные изменения в ветке.**
4 из 8 файлов не относятся к задаче кастомных колонок:
- `oauth_controller.go`: смена уровня логирования + удаление проверки `url == ""`
- `media/model.go`: удаление workspace ID из switch
- `workspace/queries.go`: удаление feature-flag
- `render/error.go`: смена уровня логирования

Рекомендация: вынести в отдельный коммит или MR. Смешивание усложняет ревью, откат и git blame.

**5. Удаление проверки `url == ""` в `saveAvatar`.**
Раньше пустой `url` — no-op. Теперь `http.NewRequest` с пустым URL вернёт ошибку. Нужно убедиться, что все вызывающие стороны гарантируют непустой URL.

### Minor (6)

**6. Неиспользуемый импорт `subtitle`.** Поле `Subtitles subtitle.Subtitles` добавлено в структуру `Media`, но в SQL `subtitles` не выбирается — только `subtitles_enabled`. Поле всегда будет пустым.

**7. Мёртвое поле `PlayerID`.** В структуре `Media` есть `PlayerID *uuid.UUID`, но в SQL выбирается только `player_name`. Поле всегда `nil` в JSON.

**8. Дублирование `EventID`.** Поле `EventID *uuid.UUID` с JSON-ключом `event_id` есть и в `Media`, и в `Model` — на разных уровнях вложенности. Может запутать фронтенд.

**9. Неочевидная сортировка по jsonb.** Сортировка по `tags`, `chapters`, `privacy_domains` — лексикографическая по JSON-представлению. Результаты неинтуитивны для пользователя.

**10. Тип `[]string` для jsonb-полей.** `Tags` и `PrivacyDomains` десериализуются из `jsonb_agg`. Если агрегация вернёт `null`-элементы — в массиве окажутся пустые строки. Фильтр `WHERE T.deleted_at IS NULL` должен это предотвращать, но стоит проверить.

**11. Качество коммитов.** 13 коммитов с сообщениями `up`, `fixes`, `rm unnecessary`, `up scan`. Перед merge нужен squash с информативным описанием.

## Чему учит этот кейс

**SQL-запросы — отдельная ось ревью.** 3 из 5 Important-замечаний — про SQL: LATERAL JOIN до пагинации, лишние JOIN в count-запросе, подстановка пользовательского ввода в ORDER BY. Стандартный Go-чеклист (контракты, ошибки, конкурентность) эти проблемы не ловит. Для проектов, где основная логика в SQL-строках внутри Go-кода, нужна отдельная секция в чеклисте.

**Scope ветки — не мелочь.** Половина файлов в diff не относилась к задаче. Это затрудняет ревью (нужно разбираться, что к чему), откат (нельзя откатить фичу без отката несвязанных правок) и git blame (изменения привязаны к неправильной задаче). Ревьюер должен явно проверять scope.

**Мёртвые поля в контракте API.** Три поля в структуре `Media` добавлены, но не заполняются из SQL. Фронтенд видит `null`/пустые значения и не знает — это баг или так задумано. Чеклист «нет silent contract break» не ловит обратную проблему — добавление полей, которые никогда не содержат данных.

**Качество коммитов видно при ревью.** 13 коммитов `up`/`fixes` не дают понять историю изменений. Squash с нормальным описанием перед MR — не формальность, а инструмент для ревьюера.

**Kaiten даёт контекст.** Чтение карточки #56769756 через MCP дало список полей, группы колонок, технические требования — без этого ревью было бы «смотрим код в вакууме».
