# Кейс: диагностика диска и RAID-аудит

Инженер вставил текст алерта в чат Cursor. За одну сессию AI диагностировал причину заполнения диска (deleted-but-open files от зависшего ffmpeg, ~103 GB), а затем параллельно собрал данные о деградированных RAID-массивах на двух серверах и выдал конкретные модели дисков под замену. Всё через SSH, без ручного подключения к серверам.

## Ситуация

Алерт из мониторинга:

```
The root partition is more than 75% full on the instance ru-msk-ost-prod-encoder-2, current: 75%
```

Сервер — продовый энкодер, транскодирует видео. Инженер скопировал текст алерта в Cursor и попросил разобраться.

## Что сделал AI

### Роутинг и подключение

AI автоматически выбрал скилл `linux-engineer` по содержимому алерта. MCP-сервер `kinescope-ops-loc` был в ошибке — AI переключился на прямой SSH:

```bash
ssh -o ConnectTimeout=10 ru-msk-ost-prod-encoder-2
```

### Расхождение df vs du — deleted-but-open files

Первый шаг — стандартная пара команд для анализа дискового пространства:

```bash
df -hT /
```

```
/dev/md127p1  ext4  213G  150G  53G  74%  /
```

```bash
du -xh --max-depth=1 /
```

Суммарно `du` показал ~25 GB. Расхождение **150 GB (df) vs 25 GB (du)** — классический признак удалённых файлов, которые всё ещё удерживаются открытыми процессами. `df` считает по метаданным файловой системы (блоки заняты), `du` обходит дерево каталогов (удалённые файлы не видны).

Следующая команда — `lsof +L1`, которая показывает файлы с нулевым link count (удалены из каталога, но дескриптор открыт):

```bash
lsof +L1
```

Результат — процесс `kinescope-ffmpeg-custom` (PID 3001229, user `kinescope`), работавший **23 часа**, держал открытыми 7 удалённых файлов в `/tmp/transcoder/`:

| FD | Файл (deleted) | Размер |
|----|-----------------|--------|
| 3 | `source/chunk_0.mp4` | ~45.6 GB |
| 5 | `assets/0/2160/2160.mp4` | ~22.4 GB |
| 6 | `assets/0/1440/1440.mp4` | ~15.7 GB |
| 7 | `assets/0/1080/1080.mp4` | ~8.2 GB |
| 8 | `assets/0/720/720.mp4` | ~5.8 GB |
| 9 | `assets/0/480/480.mp4` | ~3.3 GB |
| 4 | `assets/0/360/360.mp4` | ~2.1 GB |

**Итого ~103 GB** удалённых, но не освобождённых данных — транскодинг 4K видео в 6 разрешений.

### Реальное использование диска

Без учёта deleted files диск использовался скромно:

| Каталог | Размер | Основное |
|---------|--------|----------|
| `/var` | 8.2 GB | `/var/lib` — 7.2 GB, логи — 921 MB |
| `/usr` | 5.9 GB | — |
| `/tmp` | 1.6 GB | `/tmp/asr` — 1.4 GB |
| `/home` | 725 MB | — |

### Варианты действий

AI работал в copilot-режиме — собрал данные и предложил варианты, а не выполнил kill самостоятельно:

1. **Подождать** — если ffmpeg ещё реально кодирует, процесс завершится и ~103 GB освободятся.
2. **Проверить, жив ли процесс** — посмотреть CPU и рост выходных файлов. Если CPU ~0 и файлы не растут — процесс завис.
3. **Kill процесса** (только с подтверждением инженера) — `kill 3001229` моментально освободит ~103 GB.

23 часа транскодинга были отмечены как подозрительно долго даже для 4K — вероятный зависший процесс.

### RAID-аудит: параллельный сбор с двух серверов

В продолжение той же сессии инженер попросил проверить диски на двух серверах с деградированными RAID-массивами. AI запустил два SSH-подключения параллельно и собрал данные одновременно.

На обоих серверах SSH вернул `Host key verification failed` — AI автоматически добавил `-o StrictHostKeyChecking=accept-new` и повторил подключение.

На каждом сервере выполнялась цепочка:

```bash
cat /proc/mdstat
mdadm --detail /dev/md*
smartctl -a /dev/<disk>
mdadm --examine /dev/<disk>
lsblk -o NAME,SIZE,TYPE,FSTYPE,MOUNTPOINT
```

RAID: md0 — RAID1, **DEGRADED** `[2/1] [U_]`

| Слот | Устройство | Модель | Serial | SMART | Состояние |
|------|-----------|--------|--------|-------|-----------|
| 0 | nvme1n1 | INTEL SSDPE2KE020T7 (P4610 2TB NVMe) | BTLE817503CD2P0DGN | PASSED (wear 2%, 0 ошибок, 50805 ч) | OK |
| 1 | nvme0n1 | — | — | — | Физически отсутствует (No such device) |

Второй NVMe полностью выпал из системы — устройство не видно ядром.

#### ru-msk-ixlrt-bcp-1

- md0 (boot, 5 GB) — OK `[UU]`
- md1 (root, 218 GB) — OK `[UU]`
- md2 (data, 14 TB) — **FAILED** `[2/1] [U_]`, State: `clean, FAILED`

| Устройство | Модель | Serial | Размер | SMART | В RAID |
|-----------|--------|--------|--------|-------|--------|
| sda | INTEL SSDSC2KB240G8 | BTYF82820N6X240AGN | 240 GB SSD | PASSED | md0, md1 (OK) |
| sdb | INTEL SSDSC2KB240G8 | BTYF82820N84240AGN | 240 GB SSD | PASSED | md0, md1 (OK) |
| sdd | TOSHIBA MG07ACA14TE | 6230A08ZFBWG | 14 TB HDD 7200rpm | PASSED | superblock md2, но не в массиве |
| sde | TOSHIBA MG07ACA14TE | 6230A0HHFBWG | 14 TB HDD 7200rpm | PASSED | superblock md2, но не в массиве |

md2 в статусе FAILED: `sdc` исчез из `/dev/`, а `sdd` и `sde` с валидными суперблоками md2 не подключены к работающему массиву. Данные доступны через LVM (`/dev/mapper/vg0-lv--0` -> `/data`, 3.4T из 13T занято), но массив держится ненадёжно.

## Результат

Инженер получил за одну сессию:

1. **Причину алерта** — не реальное переполнение, а ~103 GB deleted-but-open files от зависшего ffmpeg.
2. **Варианты действий** с оценкой рисков — без автоматического kill, решение за инженером.
3. **Готовый заказ на замену дисков:**

| Сервер | Что менять | Модель для заказа |
|--------|-----------|-------------------|
| kinescope-postgres-3 | NVMe slot 0 (nvme0n1) | Intel P4610 2TB NVMe (SSDPE2KE020T7) |
| ru-msk-ixlrt-bcp-1 | HDD в md2 (sdc пропал) | Toshiba MG07ACA14TE 14TB 7200rpm |

4. **Предупреждение** о нестабильной ситуации с md2 на bcp-1 (sdd/sde с валидными суперблоками, но вне массива) — требуется план восстановления перед заменой.

## Чему учит этот кейс

**df vs du расхождение — не аномалия, а паттерн.** Если `du` показывает значительно меньше, чем `df`, первое действие — `lsof +L1`. На серверах с долгоживущими процессами (транскодинг, стриминг, ETL) это частая причина.

**AI в copilot-режиме для деструктивных действий.** AI предложил kill как вариант, но не выполнил — ждал подтверждения инженера. Для ops-задач это правильное поведение: собрать данные и показать варианты, не действовать автоматически.

**Параллельный сбор данных с нескольких серверов.** Два SSH-вызова одновременно, адаптация к host key verification — экономия времени на рутинном сборе с одинаковым набором команд.

**Полная цепочка от алерта до заказа оборудования.** Одна сессия покрыла диагностику (df/du/lsof), аудит RAID (mdadm/smartctl), идентификацию моделей и серийников — обычно это несколько итераций между инженером и датацентром.

**Скилл как набор команд и порядок их применения.** `linux-engineer` задал последовательность: df -> du -> lsof -> анализ процесса. Без скилла модель могла бы сразу предложить «удалите большие файлы в /var/log» — типичный ответ при поверхностном анализе.

## Команды-справочник

| Задача | Команда |
|--------|---------|
| Общая занятость диска | `df -hT /` |
| Размер каталогов на корневом разделе | `du -xh --max-depth=1 /` |
| Удалённые файлы, удерживаемые процессами | `lsof +L1` |
| Размер конкретного deleted file через /proc | `ls -lh /proc/<PID>/fd/<FD>` |
| Статус RAID-массивов | `cat /proc/mdstat` |
| Детали массива | `mdadm --detail /dev/md<N>` |
| Суперблок диска (принадлежность к массиву) | `mdadm --examine /dev/<disk>` |
| SMART-статус диска | `smartctl -a /dev/<disk>` |
| Модель и серийник NVMe | `smartctl -a /dev/nvme<N>` или `nvme id-ctrl /dev/nvme<N>` |
| Блочные устройства | `lsblk -o NAME,SIZE,TYPE,FSTYPE,MOUNTPOINT` |
| Проверить CPU процесса | `top -p <PID> -bn1` или `ps -p <PID> -o %cpu,etime` |
