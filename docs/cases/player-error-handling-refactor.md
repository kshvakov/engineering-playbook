# Кейс: рефакторинг обработки ошибок плеера

Инженер написал в чат: «плеер падает в generic ошибки, не устойчив к битым чанкам». За одну сессию AI исследовал обработку ошибок в трёх медиа-драйверах (hls.js, dash.js, shaka-player), нашёл системные проблемы в 10+ файлах, формализовал единую таксономию ошибок и создал в Kaiten иерархию из 8 задач (родительская + 7 подзадач) с чеклистом acceptance criteria — от жалобы до полной декомпозиции без переключения из IDE.

Дата: 26 марта 2026.

## Ситуация

Сообщение инженера в Cursor:

> Плеер падает в generic ошибки, не различая их, что само по себе плохо, а второе — он валится туда по любому поводу и не устойчив к ошибкам. Например, если попадается кривой чанк, плеер падает с ошибкой, а ожидаемо — или «рассыпаться на кадрах» и продолжить, или просто пропускать N битых чанков.

Контекст: видеоплеер Kinescope работает с тремя медиа-библиотеками — hls.js, dash.js и shaka-player. Каждая имеет свою систему ошибок. Плеер оборачивает их в единый интерфейс через `MediaDriver`, но обработка ошибок исторически росла точечно: каждый драйвер обрабатывает ошибки по-своему, severity назначается непоследовательно, recovery-стратегии фрагментарны.

## Что сделал AI

### Исследование архитектуры ошибок

AI начал с базового класса `MediaDriver.ts` и прошёл по всей цепочке обработки ошибок:

```
MediaDriver (базовый класс, handleError)
  → HlsDriver / DashDriver / ShakaDriver (драйвер-специфичная обработка)
    → VideoController (переэмиссия, фильтрация)
      → PlayerController (верхнеуровневая логика)
        → EventBus (публичные события)
          → errorsStore (MobX, UI)
```

Для каждого уровня AI читал код, фиксировал проблемы и сравнивал поведение между драйверами.

### Находки по драйверам

**HLS.js — 4 проблемы:**

1. `FRAG_PARSING_ERROR` (битый чанк) маппится в `UnsupportedSourceError` с `severity: 'fatal'` — плеер мгновенно умирает вместо пропуска чанка.
2. `appendErrorMaxRetry` закомментирован (`// appendErrorMaxRetry: 5`) — hls.js не пытается повторить append при ошибке декодирования.
3. Сетевые ошибки в `default`-ветке `switch` получают `severity: 'fatal'` по умолчанию из `handleError`, хотя hls.js настроен на бесконечные retry для фрагментов.
4. `recoverMediaError()` вызывается для всех `MEDIA_ERROR` кроме двух конкретных — без учёта, что некоторые из них не подлежат recovery.

```typescript
// HlsDriver.ts — битый чанк = мгновенная смерть
case ErrorTypes.MEDIA_ERROR: {
  if (
    errorData.details === ErrorDetails.MANIFEST_INCOMPATIBLE_CODECS_ERROR ||
    errorData.details === ErrorDetails.FRAG_PARSING_ERROR  // <-- fatal
  ) {
    this.handleError(
      new UnsupportedSourceError(...),
      { type: 'media', path: 'hls.error' }
      // severity не указан → default 'fatal'
    );
  }
}
```

**dash.js — 2 проблемы:**

1. `isFatalError()` слишком широкая: любой `MediaError` на `<video>` элементе считается fatal, даже если dash.js внутренне восстановился.
2. Хардкоженные диапазоны кодов ошибок (`10–11`, `23–35`, `100–114`, `200–201`) — хрупко при обновлении dash.js.

```typescript
// DashDriver.ts — слишком широкий isFatalError
function isFatalError(media: HTMLMediaElement, error: ErrorEvent['error']): boolean {
  return (
    media.error instanceof MediaError ||  // <-- любой MediaError = fatal
    (typeof error === 'object' &&
      ((error.code >= 10 && error.code <= 11) || ...))
  );
}
```

**shaka-player — 1 проблема:**

Все ошибки категории `MEDIA` маппятся в `UnsupportedSourceError` — семантически неверно для транзиентных ошибок декодирования отдельных сегментов.

```typescript
// shaka-driver/utils.ts — все MEDIA = UnsupportedSourceError
case 'media':
  return [type, new UnsupportedSourceError(error.message, { cause: toCause(error, shaka) })];
```

### Находки в общей архитектуре

**`MediaDriver.handleError()`** — severity по умолчанию `'fatal'`. Если драйвер не указал severity явно, ошибка автоматически становится fatal:

```typescript
// MediaDriver.ts
this.emit(this.Events.Error, {
  ...overrides,
  severity: overrides?.severity ?? 'fatal',  // <-- default fatal
  ...
});
```

**`VideoController`** — нативные ошибки `<video>` элемента не пробрасываются в EventBus (обработчик сбрасывает счётчики, но не эмитит событие):

```typescript
// VideoController.ts
mediaListener.on('error', () => {
  this.loadTaskCounter.reset('default');
  this.loadTaskCounter.reset('wait');
  // ← ошибка молча проглочена
});
```

**`errorsStore`** — UI обновляет `current` только для fatal ошибок. Warning-ошибки (деградация) никак не отражаются в интерфейсе:

```typescript
// object-stores.ts
add(error) {
  this.update((prev) => ({
    last: [prev.last[1], prev.last[2], error],
    current: error.severity === 'fatal' ? error : prev.current,
  }));
},
```

### Формализация: от проблем к таксономии

После исследования AI предложил единую таксономию ошибок, не зависящую от конкретного драйвера:

| Категория | Примеры | Recovery |
|-----------|---------|----------|
| Сегменты/чанки | Битый чанк, ошибка декодирования | Пропуск, retry, `recoverMediaError` |
| Сеть | Таймаут, 404, потеря соединения | Retry с backoff, переключение CDN |
| DRM | Лицензия, ключ, неподдерживаемая система | Retry лицензии, fallback |
| Манифест | Парсинг, пустой манифест, несовместимые кодеки | Retry, fallback на другой формат |
| Устройство/среда | Нехватка памяти, неподдерживаемый кодек | Снижение качества, уведомление |
| Escalation | N warning за T секунд | Повышение до fatal |

Ключевой принцип: **ошибка по умолчанию — warning, а не fatal**. Fatal только после исчерпания recovery-стратегий или при явно неустранимых проблемах (неподдерживаемый кодек, отсутствие DRM-системы).

**Чеклист Acceptance Criteria** на родительской карточке:

- Единая таксономия ошибок, независимая от драйвера
- Каждый тип ошибки имеет recovery-стратегию

- Ошибки логируются с полным контекстом
- Не сломана обратная совместимость public Error event API

## Фаза 2: от задач к дизайну UI и телеметрии

Через некоторое время после аудита инженер вернулся к задаче с новым запросом:

> Нужно эти ошибки также отображать пользователю в UI — как минимум это облегчает диагностику, когда присылают просто скрин. Второе — надо проработать систему отправки сообщений для диагностики сбоев. Бэкенд для приёмки можем сделать любой. Логи храним в ClickHouse, метрики можем сохранять как в Prometheus, так и агрегировать в materialized view.

Задача требовала не плана реализации, а **проектирования** — выбора из альтернатив. AI использовал brainstorming-скилл: серию вопросов с вариантами ответов, по одному за раз.

## Чему учит этот кейс

**Проблема не в одном месте, а в паттерне.** Первое сообщение про «битый чанк» наводит на мысль о точечном фиксе в hls.js. Системный анализ показал, что та же проблема (неправильная классификация severity, отсутствие recovery) повторяется во всех трёх драйверах и на всех уровнях архитектуры. Точечный фикс замаскировал бы проблему в одном месте.

**AI как архитектурный аудитор.** Кросс-библиотечный анализ — сравнение обработки ошибок в hls.js, dash.js и shaka одновременно, с учётом того, как каждый драйвер взаимодействует с общей архитектурой — задача, которая вручную заняла бы часы. AI прошёл 10+ файлов и выстроил полную картину за 20 минут.

**От исследования к задачам, а не к коду.** Соблазн — сразу начать чинить `FRAG_PARSING_ERROR`. Но инженер направил работу в формализацию: сначала понять масштаб, потом декомпозировать, потом реализовывать. Результат — не один патч, а структурированный roadmap из 7 задач.

**Иерархия задач через MCP за одну сессию.** `create_card` (8 карточек) + `add_child_card` (7 связей) + `create_checklist` (7 пунктов AC) — всё через Jarvis, без переключения в браузер. Каждая подзадача содержит описание проблемы, текущее поведение, предлагаемое решение и ключевые файлы.

**Severity по умолчанию определяет устойчивость.** Одна строка `severity: overrides?.severity ?? 'fatal'` в `MediaDriver.handleError()` означает, что любая ошибка без явного override убивает плеер. Инверсия default (warning вместо fatal) — архитектурное решение, которое меняет поведение всей системы.

**Формализация уточняется в диалоге.** Первый запрос — про битые чанки. Второй — «это не только hls.js». Третий — «это общая задача по ошибкам, DRM — подзадача». Четвёртый — «добавь ошибки устройства и какие ещё бывают у плееров». Каждое уточнение расширяло scope, и AI адаптировал анализ.

**Brainstorming — не монолог, а серия развилок.** Вместо того чтобы предложить готовый дизайн, AI задал 8 вопросов с вариантами ответов. Каждый вопрос — точка принятия решения. Инженер выбирает, AI фиксирует и строит следующий вопрос на основе ответа. Это быстрее, чем писать RFC, и точнее, чем «сделай как считаешь нужным».

**Дизайн по частям с подтверждением.** После brainstorming AI оформил дизайн не одним документом, а секциями по 200–300 слов. После каждой — «это выглядит правильно?». Инженер подтверждал или корректировал. Так ошибки ловятся на ранней стадии, а не после 2000 слов дизайна.

**Трёхуровневая иерархия задач.** Фаза 1 создала эпик + подзадачи. Фаза 2 добавила под-эпик внутри эпика + свои подзадачи. Итого: эпик (#62530972) → под-эпик (#62531197) → подзадачи. Kaiten через MCP поддерживает это нативно через `add_child_card`.

**Продолжение работы между сессиями.** Фаза 2 началась в новой сессии. Контекст восстановлен через summary предыдущего диалога + Kaiten-карточки. Задачи в трекере — не только для исполнителей, но и для AI: они хранят решения, которые иначе потерялись бы между сессиями.
