# Кейс: аудит Proxmox -- VM и ёмкость

Реальный пример использования AI-ассистента (Cursor + MCP-сервер `kinescope-ops-loc`) для инвентаризации виртуальных машин и capacity-анализа пяти Proxmox-серверов. От первого вопроса в чате до сохранённого отчёта с алертами -- один диалог, без SSH-доступа, без скриптов.

Дата: 3 февраля 2026.

## Ситуация

Инженер написал в чате Cursor: "покажи Список QEMU VM по всем Proxmox-серверам в виде таблицы". За этим стояла задача: быстро понять, что где крутится, сколько ресурсов выделено и есть ли проблемы с ёмкостью.

Инфраструктура: 5 Proxmox-серверов в одном ДЦ (MSK), от 8 до 80 CPU cores на хост, суммарно 72 QEMU VM.

## Что сделал AI

### Попытка 1: скрипт (отвергнута)

AI предложил bash-скрипт: `ssh` по списку хостов -> `pvesh get /cluster/resources --type vm --output-format json` -> `jq` для парсинга -> `column -t` для форматирования. Инженер сказал "через подключенные tools" -- ему не нужен был скрипт, он хотел использовать уже настроенное MCP-подключение.

### Попытка 2: MCP tool `pve_list_vms`

AI нашёл в MCP-сервере `kinescope-ops-loc` tool `pve_list_vms`, вызвал его и получил данные: **72 VM, 0 errors**. Результат -- таблица с колонками: dc, server, node, vmid, name, status, cpu, ram_gib, disk_gib, addr.

### От списка VM к capacity-анализу

Запрос эволюционировал в диалоге:

1. **"сделай таблицу с группировкой по server и status"** -- AI агрегировал данные в сводку:

| server | status | vm_count | cpu_total | ram_gib_total |
|---|---|---:|---:|---:|
| developers | running | 18 | 60 | 91.0 |
| developers | stopped | 5 | 9 | 16.0 |
| stage | running | 32 | 108 | 111.8 |
| stage | stopped | 17 | 53 | 58.0 |

2. **"покажи детальную информацию по VM Kirill"** -- AI вывел карточку VM: vmid 100, 4 vCPU, 6 GiB RAM, 8 дисков (scsi0-scsi7) на storage `local-lvm`, суммарно 126 GiB. Все диски с `cache=writeback, discard=on, iothread=1, ssd=1`.

3. **"покажи список хостов и информацию по ним"** -- сначала AI попытался агрегировать информацию из VM-данных (assigned ресурсы), но этого было недостаточно -- не хватало реальных данных по железу хоста. Инженер уточнил "выбери через list_hosts", AI нашёл tool `pve_list_hosts` и получил данные с CPU/RAM/Disk хоста + storage'и.

### Troubleshooting MCP-подключений

Опрос 5 серверов не сработал с первого раза. Итеративная отладка заняла 5 попыток:

| # | Что произошло | Проблемный сервер | Ошибка | Действие |
|---:|---|---|---|---|
| 1 | 2 хоста ok, 1 error | ru-msk-ost-infra-pve-2 | `context deadline exceeded (Client.Timeout exceeded while awaiting headers)` | Инженер увеличил таймаут |
| 2 | Таймаут ушёл, появился 401 | ru-msk-ost-infra-pve-2 | `http status 401` | Инженер проверил креды |
| 3 | 3 хоста ok, 0 errors | -- | -- | Запрос всех серверов |
| 4 | 3 хоста ok, 1 error | ru-msk-ixlrt-infra-pve-1 | `parse "https://https://10.3.8.2:8006/...": invalid URL escape` | Инженер исправил конфиг (двойной `https://`) |
| 5 | 4 хоста, затем 5 -- все ok | -- | errors=0 | Финальный опрос |

Каждая итерация давала диагностическую информацию: таймаут указал на сетевую проблему, 401 -- на авторизацию, ошибка парсинга URL -- на опечатку в конфиге. AI не пытался угадывать причину, а показывал точный текст ошибки.

### Пороги и алерты

AI предложил набор порогов и проверил каждый хост/storage по ним:

| Ресурс | Порог |
|---|---|
| CPU (host) | `cpu_free_% < 10%` |
| RAM (host) | `ram_free_% < 10%` или `ram_free < 4 GiB` |
| Disk (host) | `disk_free_% < 10%` или `disk_free < 50 GiB` |
| Storage (active) | `avail_% < 10%` или `avail < 100 GiB` |
| vCPU overcommit | `vcpu_assigned > cpu_total` (отдельно для running и total) |

## Результат

Инженер получил три уровня данных в одном диалоге:

**1. Хосты с проблемами (5 из 5 серверов имели хотя бы один алерт):**

| server | node | алерты |
|---|---|---|
| developers | msk-ixlrt-dev-1 | overcommit vCPU: 60 running на 12 cores |
| ru-msk-ixlrt-infra-pve-1 | ru-msk-ixlrt-infra-pve-1 | мало RAM (5.2% free -- 13.1 GiB из 251.4), overcommit vCPU |
| ru-msk-ost-infra-pve-2 | ru-msk-ost-infra-pve-2 | мало RAM (11.9% free -- 1.9 GiB из 15.6) |
| ru-msk-ost-pve-1 | ru-msk-ost-pve-1 | мало disk host (24.1% free -- 7.5 GiB из 31.3) |
| stage | pve26 | overcommit vCPU: 108 running на 12 cores |

**2. Storage с проблемами:**

| server | storage | type | avail/total | avail_% |
|---|---|---|---|---:|
| developers | local-lvm | lvmthin | 96.3/758.3 GiB | 12.7 |
| ru-msk-ixlrt-infra-pve-1 | local-zfs | zfspool | 156.4/3457.0 GiB | 4.5 |
| ru-msk-ost-pve-1 | local | dir | 5.9/31.3 GiB | 18.9 |

**3. VM на проблемных storage'ах** -- для каждого проблемного storage AI вывел список VM с привязкой дисков. Например, на `infra-pve-1/local-zfs` (4.5% avail, 156 GiB из 3457) -- 31 VM, из которых `virtbackuppg-off` (stopped) занимает 1026 GiB, `proxy` -- 760 GiB, `prometheus` -- 715 GiB.

Финальный отчёт AI сохранил в файл `low_cap.md` по просьбе инженера.

## Чему учит этот кейс

**Запрос уточняется в диалоге.** Начальный вопрос "покажи список VM" превратился в полноценный capacity-audit с алертами. AI не пытался угадать итоговую цель -- каждый следующий шаг инициировал инженер.

**MCP-tools дают прямой доступ к инфраструктуре.** Инженеру не нужно было писать скрипты, настраивать SSH, ставить зависимости. `pve_list_vms` и `pve_list_hosts` вернули структурированные данные, AI отформатировал и агрегировал их.

**Ошибки подключений -- часть процесса.** Из 5 серверов с первой попытки ответили 2. Диагностика через MCP (точный текст ошибки от каждого сервера) позволила итеративно починить: таймаут -> увеличить timeout, 401 -> проверить авторизацию, двойной `https://` -> исправить конфиг. Без MCP-tool этот цикл потребовал бы ручного дебага на каждом сервере.

**AI считает, человек решает.** AI предложил пороги (RAM < 10%, Storage < 10%) и подсветил проблемы. Но решение, что делать с overcommit vCPU 60/12 на developers или с 1026 GiB stopped-бэкапом на infra-pve-1 -- за инженером.

**Сохраняемость.** Результат диалога -- не скриншот чата, а структурированный markdown-файл с таблицами, который можно положить в репозиторий, отправить коллегам или использовать как baseline для следующего аудита.

## Полезные MCP-инструменты

| tool | что делает | пример запроса в чате |
|---|---|---|
| `pve_list_vms` | Список всех QEMU VM по всем Proxmox-серверам из конфига. Возвращает vmid, name, status, cpu, ram, диски, node, server, dc | "покажи список VM по всем Proxmox-серверам" |
| `pve_list_hosts` | Информация по хостам Proxmox: CPU/RAM/Disk (физ.), storage'и, назначенные vCPU из VM | "покажи информацию по хостам" / "выбери через list_hosts" |

MCP-сервер: `kinescope-ops-loc`. Оба tool-а поддерживают фильтрацию по `server_name` и `dc`.
