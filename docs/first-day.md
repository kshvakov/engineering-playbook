# Первый рабочий день с AI

Пошаговый сценарий: от открытия проекта до push. Рассчитан на то, что [установка и настройка](setup.md) уже сделаны.

## 1. Открыть Cursor и проект

1. Запустите Cursor.
2. **File -> Open Folder** и выберите корень репозитория (тот, где лежит `go.mod`, `package.json` или аналог).
3. Дождитесь индексации; при первом открытии большого монорепо это может занять время.

## 2. Понять структуру проекта

В чате агента задайте запрос на обзор:

```text
Расскажи о структуре проекта: основные пакеты, где точка входа, где тесты и конфиги.
```

Проверьте ответ по факту: откройте пару указанных путей в дереве файлов. Если ассистент «додумал» несуществующие каталоги, уточните: «Перечисли только то, что реально есть в файловой системе».

## 3. Взять задачу из Kaiten

Просто попросите:

```text
Покажи мои задачи в Kaiten. Выбери одну небольшую на первый день.
```

Cursor сам обратится к Kaiten через MCP, найдёт ваши карточки и покажет список. Выберите задачу с понятным scope (исправление, мелкая правка, локальная фича), а не «переписать половину сервиса».

Зафиксируйте **ID карточки** (число в URL Kaiten, например `…/card/62353324` -> `62353324`). Он понадобится для ветки.

## 4. Формализовать задачу

Скилы подхватываются автоматически, если они установлены. Просто попросите:

```text
Формализуй эту задачу: цель, входные данные, вопросы P0-P2, критерии готовности, что вне scope. Не заполняй пробелы домыслами.
```

Сохраните результат себе в заметку или в черновик описания MR: по нему вы сверитесь после реализации.

## 5. План

Когда формализация есть, попросите план:

```text
Составь план реализации: шаги, файлы/пакеты, риски, где нужны тесты. План должен быть выполнимым за один рабочий день.
```

Не переходите к массовым правкам, пока план не согласован с вами: прочитайте и отрежьте лишнее.

## 6. Ветка, привязанная к задаче

Можно создать ветку руками:

```bash
git fetch origin
git checkout -b feature/kaiten-62353324 origin/main
```

А можно попросить ассистента:

```text
Создай ветку от master и привяжи к задаче
```

Cursor сам сделает checkout, создаст ветку с правильным суффиксом `kaiten-{id}` и запушит. Подставьте свой тип ветки (`fix/`, `feature/`) — или ассистент подберёт по контексту.

## 7. Выполнить план по шагам

Порядок работы:

1. Вставьте в чат план целиком или по частям.
2. Формулируйте шаг явно: «Сделай только шаг 2 из плана, не трогай остальное».
3. После каждого шага просматривайте diff: нет ли лишних файлов и «улучшений» вне задачи.

Если план большой, используйте скил **executing-plans**: он выполняет задачи пакетами с чекпоинтами.

## 8. Верификация

Попросите ассистента проверить сборку:

```text
Проверь что всё собирается и тесты проходят. Не утверждай успех без вывода команд.
```

Cursor сам запустит `go build`, `go test`, `eslint` или что нужно для вашего стека. Скил **verification-before-completion** как раз про то, чтобы не верить словам «всё собралось» без лога.

Если хотите запустить руками:

**Go**

```bash
go build ./...
go test ./...
```

**Node**

```bash
npm run build
npm run lint
```

## 9. Коммит и push

Сообщения коммитов для `gitlab.kinescope.dev` пишите **на русском**, по смыслу изменения:

```bash
git add -p
git commit -m "Исправлен расчёт таймаута для ретраев (kaiten-62353324)"
git push -u origin feature/kaiten-62353324
```

Дальше создайте merge request в GitLab и прикрепите ссылку на карточку Kaiten в описании.

## 10. Советы на первый день

- **Начните с малого** — одна карточка, один пакет, минимальный diff. Так вы быстрее поймёте, как агент ведёт себя в вашем коде.
- **Проверяйте всё инструментами** — сборка, тесты, линтер. AI ошибается так же, как человек, только увереннее формулирует.
- **Не доверяйте вслепую** — читайте diff, сомневайтесь в «рефакторинге заодно», переспрашивайте, если ответ противоречит коду.
- **Держите контекст узким** — открывайте нужные файлы, ссылайтесь на пути; так меньше шанс, что модель угадает несуществующую структуру.
- **MCP — не магия** — если Kaiten или БД не отвечают, проверьте токены, VPN и конфиг MCP.

После первого дня имеет смысл пройти [Полный цикл работы](workflow/overview.md) и [Обязательные скилы](must-have-skills.md), чтобы выровнять процесс с командой.
