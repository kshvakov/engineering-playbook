# MCP: зачем и как это работает

## Что такое MCP

**MCP (Model Context Protocol)** — протокол, через который ассистент в IDE получает доступ к внешним системам: таск-трекеру, мессенджеру, базам данных. Вы не вызываете никакие API руками — вы просто пишете на обычном языке, а Cursor сам решает, к какому сервису обратиться.

## Без MCP: работа «в вакууме»

Если интеграций нет, типичный цикл такой:

- вы копируете описание из таск-трекера в чат;
- вручную ищете обсуждение в мессенджере и вставляете цитаты;
- открываете админку БД или отдельный клиент, чтобы посмотреть схему или прогнать запрос;
- снова переключаетесь между вкладками и рискуете потерять нить контекста.

Ассистент не видит живых систем и опирается только на то, что вы явно вставили в диалог.

## С MCP: одна точка входа — IDE

С подключёнными MCP-серверами вы **просто говорите что нужно**, а ассистент сам обращается к нужным системам. Не нужно знать имена инструментов, параметры или порядок вызовов — Cursor разбирается сам.

## Как это выглядит на практике

Вот реальные примеры из рабочих сессий:

| Что пишете в чат | Что происходит за кулисами |
|---|---|
| «создай задачу на доработку botdetect и привяжи ветку» | Cursor находит нужную доску в Kaiten, создаёт карточку, делает `git checkout -b` с правильным суффиксом, пушит ветку |
| «покажи мои задачи которые можно закрыть» | Cursor читает подзадачи из Kaiten, сопоставляет со статусом работы в репозитории, выдаёт таблицу с рекомендацией |
| «проверь схему таблицы botdetect.events в ClickHouse» | Cursor обращается к ClickHouse, получает структуру таблицы, показывает поля и типы |
| «pq: permission denied for schema botdetect» | Cursor анализирует коннекты к PostgreSQL, находит причину (отсутствие GRANT USAGE), предлагает миграцию |
| «найди обсуждения по этой задаче в Пачке» | Cursor ищет упоминания номера карточки в мессенджере, возвращает релевантные сообщения |

Обратите внимание: ни в одном примере нет имён инструментов или параметров. Вы описываете **что хотите**, а не **как это сделать**.

## Архитектура (упрощённо)

```text
Вы (обычный текст в чате)
        ↓
   Cursor (выбирает нужный MCP-сервер)
        ↓
   MCP-сервер (обращается к сервису)
        ↓
   Kaiten / Пачка / PostgreSQL / ClickHouse
```

Разные сервисы разнесены по разным MCP-серверам — так проще ограничивать права и настраивать доступ.

## Какие MCP-серверы мы используем

| Сервер | Что через него доступно |
|--------|------------------------|
| **Jarvis** | Kaiten (задачи, стадии, карточки) и Пачка (каналы, треды, поиск обсуждений) |
| **kinescope-dev-mcp** | PostgreSQL и ClickHouse: схемы, метаданные, выполнение SQL, диагностика |
| **GitKraken (расширение GitLens)** | Git-контекст из IDE |

Настройка серверов описана в [setup.md](../setup.md). Детали по сценариям: [таск-трекер](task-tracker.md), [мессенджер](messenger.md), [базы данных](databases.md).

## Что важно понимать

1. **Вам не нужно знать внутреннее устройство MCP** — вы просто говорите что нужно. Конкретные инструменты и их параметры могут меняться, но подход остаётся: естественный язык в чате.
2. **Политика безопасности** — особенно для БД: по умолчанию чтение, запись требует подтверждения.
3. **Именование веток** — суффикс `kaiten-{card_id}` в имени ветки связывает push и MR с карточкой автоматически.

MCP не заменяет код-ревью и здравый смысл, но снимает трение между IDE и корпоративной инфраструктурой. Вместо переключения между вкладками браузера — одна сессия в Cursor.
