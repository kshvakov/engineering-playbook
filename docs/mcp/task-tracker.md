# Таск-трекер (Kaiten) из IDE

Через MCP-сервер **Jarvis** ассистент в Cursor получает доступ к Kaiten. Вы не вызываете никакие API — просто пишете в чат что нужно сделать с задачами.

## Что можно делать и как это выглядит

| Что хотите | Что пишете в чат | Что происходит |
|---|---|---|
| Посмотреть свои задачи | «покажи мои задачи в работе» | Cursor находит ваши карточки и показывает список со статусами |
| Создать задачу | «создай задачу на доработку botdetect» | Cursor сам находит нужную доску, создаёт карточку, возвращает ID |
| Создать задачу + ветку | «создай задачу и привяжи к ней ветку от master» | Cursor создаёт карточку, делает `git checkout -b feature/...-kaiten-{id}`, пушит |
| Узнать что можно закрыть | «покажи какие задачи можно закрыть» | Cursor читает подзадачи, сопоставляет со статусом работы, выдаёт рекомендацию |
| Перевести статус | «двигай их в in progress на меня» | Cursor переводит карточки на нужную стадию |
| Посмотреть детали задачи | «что в задаче 62353324?» | Cursor показывает описание, статус, ссылки, комментарии |
| Полный контекст | «посмотри карточку и найди обсуждения в Пачке» | Cursor читает карточку + ищет упоминания в мессенджере |

## Именование веток и привязка к карточке

Соглашение: ветка содержит суффикс **`kaiten-{card_id}`**, где `card_id` — числовой идентификатор карточки из URL Kaiten.

Примеры:

- `feature/botdetect-filters-kaiten-62438030`
- `fix/player-timeout-kaiten-60966568`

Автоматика привязывает push и merge request к карточке по этому суффиксу. Если затронуты два репозитория (бэкенд и фронтенд), используйте **одно и то же имя ветки** в обоих.

## Типовой workflow

1. Вы формулируете намерение: «создай задачу на доработку botdetect и привяжи ветку».
2. Cursor создаёт карточку в Kaiten, берёт `card_id`, создаёт ветку с суффиксом, пушит.
3. Вы работаете в ветке, коммитите, пушите.
4. По готовности: «покажи какие задачи можно закрыть» — Cursor сопоставляет статус задач с тем, что сделано.

Всё это — одна сессия в IDE. Не нужно открывать Kaiten в браузере, копировать ID, вручную создавать ветки.

## Реальный пример из работы

Одна фраза в чате:

> «Старые задачи все закрыты, создай отдельную новую и к ней привяжи ветки. Старые ветки смерджены, делай pull и ветки от мастера»

Что сделал Cursor:
1. Нашёл доску в Kaiten
2. Создал карточку с описанием
3. Сделал `git pull` в двух репозиториях (бэкенд и фронтенд)
4. Создал ветки от актуального master в обоих
5. Запушил ветки
6. Вернул ссылку на карточку и имена веток

Ещё пример:

> «Нужно под это создать задачу и привязать к ней ветки, ветки сделай от мастера и кроме kaiten-N там должен быть говорящий slug»

Cursor создал задачу и ветку `feature/botdetect-filters-kaiten-62438030` — с осмысленным slug и привязкой к карточке.

## Перекрёстные ссылки

Когда ассистент читает карточку Kaiten, он автоматически извлекает ссылки из описания и комментариев: на GitLab, Figma, Пачку, другие карточки. Ссылки на Kaiten дополняются заголовком и статусом, чтобы сразу видеть — задача открыта или закрыта.

Важно: **ссылки на `kinescopers.kaiten.ru` требуют авторизации** — ассистент не может открыть их как обычную веб-страницу, но через MCP получает полный доступ к содержимому карточки.

## Иерархия задач: родительская + подзадачи + чеклист

Для крупных задач, которые декомпозируются на несколько направлений, AI может за одну сессию создать полную иерархию в Kaiten:

1. **Родительская карточка** — общее описание проблемы, целевая архитектура, acceptance criteria.
2. **Подзадачи** (`create_card` + `add_child_card`) — каждая самодостаточна: описание проблемы, текущее поведение, предлагаемое решение, ключевые файлы.
3. **Чеклист AC** (`create_checklist`) — на родительской карточке, проверяемые критерии готовности всей работы.

Пример из практики — рефакторинг обработки ошибок плеера:

> «Давай проработаем этот вопрос и поставим задачу в Kaiten»

Что сделал Cursor за одну сессию:

| Шаг | MCP-вызов | Результат |
|-----|-----------|-----------|
| 1 | `create_card` | Родительская карточка #62530972 с описанием проблемы и целевой архитектурой |
| 2 | `create_card` × 7 | 7 подзадач: сегменты, сеть, DRM, манифест, устройство, escalation, observability |
| 3 | `add_child_card` × 7 | Привязка подзадач к родительской |
| 4 | `create_checklist` | 7 пунктов AC на родительской: единая таксономия, recovery, UI-деградация и т.д. |

Каждая подзадача содержала структурированное описание: «Проблема → Текущее поведение → Предлагаемое решение → Ключевые файлы». Так исполнитель видит не только «что делать», но и «почему» и «где в коде».

Нюанс: `add_child_card` **не создаёт** карточку — только связывает существующие. Сначала `create_card`, потом `add_child_card` с `parent_id` и `child_id`.

### Трёхуровневая иерархия: эпик → под-эпик → подзадачи

Когда работа развивается в несколько фаз, одного уровня «родительская + подзадачи» не хватает. Kaiten поддерживает произвольную вложенность через `add_child_card`:

```
Эпик (#62530972) «Рефакторинг ошибок плеера»
├── Подзадача: Ошибки сегментов
├── Подзадача: Сетевые ошибки
├── Подзадача: DRM-ошибки
├── ...ещё 4 подзадачи фазы 1
└── Под-эпик (#62531197) «UI ошибок и телеметрия»
    ├── Подзадача: Каталог кодов ошибок
    ├── Подзадача: Fatal-экраны
    ├── Подзадача: Toast-деградация
    ├── Подзадача: Телеметрия (protobuf)
    └── Подзадача: Бэкенд приёмки
```

Как это делается через MCP:

1. Фаза 1: `create_card` (эпик) → `create_card` × N (подзадачи) → `add_child_card` × N.
2. Фаза 2: `create_card` (под-эпик) → `add_child_card` (под-эпик → эпик) → `create_card` × M (новые подзадачи) → `add_child_card` × M (подзадачи → под-эпик).

Каждый уровень может иметь свой `create_checklist` с AC. Под-эпик группирует логически связанные задачи внутри большой работы и может быть назначен отдельному исполнителю или команде.

Подробности — в [кейсе рефакторинга ошибок плеера](../cases/player-error-handling-refactor.md).

## Краткий чеклист

- Ветка содержит `-kaiten-{card_id}`.
- Для полного контекста задачи: попросите посмотреть карточку и найти обсуждения в Пачке.
- Если не знаете, на какой доске создавать — просто скажите «создай задачу», Cursor разберётся.
- Если затронуты несколько репозиториев — одно имя ветки во всех.
