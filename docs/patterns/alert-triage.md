# Триаж алертов

Текст алерта из мониторинга — готовый входной сигнал для AI. Вставляете как есть, AI выбирает роль (linux-engineer), подбирает диагностические команды и ведёт расследование по структурированному циклу.

## Когда использовать

- Disk full / high usage
- RAID degraded
- High CPU / memory / iowait
- Сервис не отвечает
- Любой алерт, требующий SSH на сервер и набора диагностических команд

## Цикл триажа

Три фазы: приём, диагностика, вывод.

### Фаза 1: приём и контекст

Вставить текст алерта в чат **как есть** — не пересказывать, не интерпретировать. AI извлекает из текста: хост, метрику, пороговое значение, текущее значение. Роутинг на скилл: для серверных алертов — linux-engineer.

### Фаза 2: диагностика

1. **Coarse** — общая картина: `df`, `free`, `top`, `cat /proc/mdstat`.
2. **Drill-down** — уточнение: `du`, `lsof +L1`, `smartctl`, `mdadm --detail`.
3. При нескольких серверах — параллельный сбор (см. паттерн «Параллельные агенты»).
4. USE-метод: Utilization, Saturation, Errors.

### Фаза 3: вывод и действие

- Структурированный отчёт: причина, масштаб, варианты действий.
- Copilot-режим: AI предлагает действия, **не выполняет** опасные без подтверждения.
- Артефакт: таблица, файл, рекомендация для заказа оборудования.

## Пример: disk full

Алерт: `root partition > 75% on ru-msk-ost-prod-encoder-2`

1. `df -hT /` — 150G used из 213G.
2. `du -xh --max-depth=1 /` — всего 25G видимых файлов.
3. Расхождение 150G vs 25G. `lsof +L1` — 103GB deleted-but-open файлов (ffmpeg).
4. Вывод: зависший процесс транскодинга держит удалённые файлы. Варианты: дождаться завершения, проверить состояние процесса, kill с подтверждением.

## Пример: RAID degraded

Алерт или запрос на проверку RAID:

1. `cat /proc/mdstat` — `[U_]` degraded.
2. `mdadm --detail /dev/md0` — какой диск missing.
3. `smartctl -i -H /dev/nvme1n1` — модель, serial, SMART status.
4. Вывод: таблица (модель, serial, состояние), что заказывать на замену.

## Антипаттерны

- **Пересказывать алерт своими словами** вместо вставки текста — AI теряет точные данные (хост, числа, пороги).
- **Сразу просить «почини» без диагностики** — AI может выполнить опасное действие без понимания причины.
- **Игнорировать расхождения в данных** (df vs du) — пропуск root cause.
- **Один запрос на 5 серверов** — лучше по одному или попарно; при необходимости — параллельные агенты.

## Связь с другими паттернами

- **Параллельные агенты:** несколько серверов с одним алертом — параллельный SSH через отдельных агентов.
- **Верификация:** после действия (kill, restart) — проверить, что место освободилось / RAID восстановился. Без свежего доказательства триаж не завершён.
