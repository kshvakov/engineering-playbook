# Привязка веток к задачам

Соглашение связывает Git-ветки и merge request'ы с карточками Kaiten, чтобы в задаче автоматически отображались ссылки на код и ревью.

## Соглашение по имени ветки

Имя ветки **заканчивается** суффиксом:

`kaiten-{card_id}`

где `{card_id}` — числовой идентификатор карточки в Kaiten (последнее число в URL карточки).

Примеры формата:

- `feature/описание-kaiten-62483776`
- `fix/null-deref-player-kaiten-61234567`

Тип префикса (`feature/`, `fix/` и т.д.) — по правилам команды; **обязателен** распознаваемый суффикс `kaiten-<id>`.

## Что даёт Jarvis MCP

При работе через интеграцию **Jarvis** (MCP для Kaiten и Пачки):

- **push** ветки с корректным суффиксом может привести к тому, что на карточке появится **ссылка на ветку**;
- **merge request** с такой же привязкой — к **ссылке на MR** на карточке.

Точное поведение зависит от настроек пространства и правил команды; суффикс — договорённый триггер для автосвязывания.

## Рабочий процесс

1. Создать или получить карточку в Kaiten (попросите ассистента: «создай задачу» или используйте существующую).
2. Взять **card_id** из URL карточки или из ответа ассистента.
3. Создать ветку локально:

   ```bash
   git checkout -b feature/краткое-описание-kaiten-{card_id}
   ```

4. Коммитить, пушить в удалённый репозиторий.
5. Открыть MR с той же веткой — карточка получит связь с ревью.

## Несколько репозиториев

Если фича затрагивает **backend и frontend** (или любые два репозитория), используйте **один и тот же** `kaiten-{id}` в имени ветки в каждом репозитории. Так по одной карточке видно согласованные ветки и MR в разных проектах.

**Пример:**

- Backend: `feature/botdetect-cleanup-goload-kaiten-62483776`
- Frontend: `feature/botdetect-cleanup-goload-kaiten-62483776`

Описательная часть может совпадать или быть согласованной; главное — **одинаковый** суффикс `kaiten-62483776`.

## Практические замечания

- Не смешивайте на одной ветке несколько несвязанных карточек: один суффикс — одна основная задача (если команда не оговорила иное).
- Если карточку перенумеровали или разбили — обновите ветку или заведите новую по правилам команды, чтобы ссылки в Kaiten не вели в пустоту.

Так дисциплина имён веток превращается в навигацию: из трекера в код и обратно без ручного копирования ссылок на каждый MR.

## Как получить card_id из ссылки

В типичном URL карточки идентификатор — **последний числовой сегмент** пути, например:

- `https://example.kaiten.ru/62483776` → `62483776`;
- `https://example.kaiten.ru/space/board/card/62483776` → `62483776`.

Если сомневаетесь, попросите ассистента: «покажи карточку по этой ссылке» — он вернёт id и детали.

## Оформление MR и описания

В заголовке или теле merge request полезно продублировать номер карточки и короткое «что сделано», чтобы ревьюеру не искать контекст. Суффикс ветки уже связывает MR с Kaiten; текст MR остаётся человеческим резюме для истории.

## Если автосвязь не появилась

Проверьте по очереди:

- суффикс ветки написан без опечаток: именно `kaiten-` и цифры;
- push ушёл в тот remote и namespace, с которыми настроена интеграция;
- для MR используется ветка с тем же именем, что пушили.

Дальше — к ответственным за Jarvis / настройки Kaiten: локально вы всё ещё можете вручную прикрепить ссылку на MR в карточке, пока разбираются с автоматикой.

## Релизы и долгоживущие ветки

Для `main`, `release/*` и интеграционных веток команды часто делают исключения: там нет суффикса Kaiten. Суффикс — для **тематических** веток под конкретную карточку, а не для постоянных потоков.
