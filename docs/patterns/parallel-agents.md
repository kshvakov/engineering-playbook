# Параллельные агенты

Паттерн из скилла **dispatching-parallel-agents**: когда у вас **две и более независимые** задачи, которые можно вести без общего изменяемого состояния и без жёсткой последовательности «сначала A, потом B».

## Зачем

Несколько несвязанных поломок (разные файлы тестов, разные подсистемы, разные баги), расследуемые строго по очереди, тратят время. Каждое расследование может идти параллельно, если домены не пересекаются.

**Принцип:** один агент — на один независимый домен проблемы. Запускать их одновременно.

## Дерево решений

Пройдите вопросы по порядку:

1. **Несколько сбоев / задач?** Если нет — достаточно одного агента.
2. **Они независимы?** Если сбои связаны (исправление одного может затронуть другие, или нужен единый контекст системы) — **не** диспатчить параллельно; один агент или последовательная работа.
3. **Можно работать параллельно?** Если есть общее изменяемое состояние (один и тот же файл, одна ветка с конфликтами, общий кэш сборки без изоляции) — либо устраните конфликт, либо идите последовательно.
4. **Диспатч:** запускайте отдельных агентов с чёткими границами.

Кратко: **много независимых задач → параллельно; связанные или с общим состоянием → нет.**

## Что получает каждый агент

- **Конкретная область:** один файл, один пакет, одна подсистема, один набор тестов.
- **Ясная цель:** например, «исправить падения в `foo_test.go`» или «привести `bar.tsx` к правилам ESLint».
- **Полный контекст в рамках области:** пути к файлам, фрагменты логов, ожидаемое поведение, ограничения («не трогать соседние модули»).
- **Ожидаемый результат:** краткое резюме: что нашли, что поменяли, как проверили.

## Пример

Нужно убрать ошибки ESLint в трёх **разных** файлах, правила не конфликтуют и файлы не зависят друг от друга:

- Агент 1: `src/api/client.ts`
- Агент 2: `src/ui/Form.tsx`
- Агент 3: `src/hooks/useAuth.ts`

После возврата агентов: прочитать сводки, убедиться, что правки не пересекаются, прогнать общую проверку (линт/сборка по политике проекта) и интегрировать.

## Когда не использовать

- **Связанные сбои:** один корневой дефект проявляется в нескольких местах — сначала локализуйте причину.
- **Общее состояние:** два агента правят один и тот же модуль без координации.
- **Нужна целостная картина:** архитектурное решение или рефакторинг сквозного потока — лучше один контекст и один план.

Параллельные агенты ускоряют **независимые** дорожки; они не заменяют анализ зависимостей между задачами.

## Как группировать независимые домены

Представьте, что падают тесты в трёх файлах с **разными** причинами:

- файл A — сценарий согласования инструмента;
- файл B — завершение батча;
- файл C — отмена (abort).

Исправление согласования не обязано менять поведение отмены: это три дорожки. Такие группы и отдаются разным агентам.

Если же все три файла падают из‑за **одной** сломанной инварианты в ядре — сначала один агент (или человек) находит корень, потом решайте, нужен ли параллелизм на уровне тестов/обвязки.

## Ограничения и интеграция

После параллельной работы **вы** остаётесь ответственным за слияние:

1. Прочитать итог каждого агента.
2. Проверить отсутствие конфликтующих правок в одних и тех же строках.
3. Запустить **полный** набор проверок репозитория (тесты, линт, сборка — по политике проекта).
4. Только после зелёного прогона — пуш и MR.

Параллельный диспатч не отменяет финальную верификацию на уровне всего изменения.

## Сравнение с одним агентом

| Ситуация | Один агент | Параллельно |
|----------|------------|-------------|
| Три независимых ESLint-файла | Дольше по wall-clock | Уместно |
| Один баг ломает три теста | Лучше сначала root cause | Обычно нет |
| Рефакторинг общего модуля | Последовательно, единый дизайн | Риск конфликтов |

Используйте дерево решений в начале документа как быстрый фильтр, а таблицу — как напоминание о типичных ошибках выбора.
