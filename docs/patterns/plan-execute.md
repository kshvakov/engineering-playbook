# Паттерн «План → Выполнение»

Самый распространённый способ довести задачу от формулировки до кода с предсказуемым качеством: сначала зафиксировать план и шаги, затем выполнять их порциями с проверками.

## Типичный поток

1. **Пользователь описывает задачу** — часто неформально, с пробелами и допущениями.
2. **ИИ формализует:** уточняет границы, при необходимости оформляет спецификацию (AC/DoD), **создаёт план** и **список задач** (todo), разбивая работу на мелкие шаги.
3. **ИИ выполняет по шагам**, отмечая прогресс (в работе / выполнено).
4. **После каждого логического шага — верификация:** сборка, линтер, тесты — что принято в проекте. Не накапливать непроверенные изменения.
5. **Коммит** с осмысленным сообщением (для репозиториев Kinescope — на русском, если так принято в команде).
6. **Пользователь смотрит результат** и даёт правки; цикл повторяется до сходимости.

Так снижается риск «сделал много сразу без проверки» и проще откатывать или перенаправлять работу.

## Два скилла

| Скилл | Роль |
|--------|------|
| **writing-plans** | Создать план **до** правок кода: цель, архитектура, стек, задачи с файлами, шаги «на один жест», что и как тестировать. План сохранять в согласованное место (например, `docs/plans/YYYY-MM-DD-<feature>.md`). |
| **executing-plans** | Выполнять готовый план **батчами** с остановками на обратную связь. |

В заголовке плана в репозитории playbook рекомендуется явно указывать под-скилл **executing-plans** для исполнителя.

## Батчи и чекпоинты

По умолчанию в **executing-plans**:

- взять **первые три задачи** из плана;
- для каждой: пометить «в работе», следовать шагам плана, прогнать указанные проверки, пометить «выполнено»;
- по завершении батча: показать, что сделано, вывести вывод проверок, сказать «готов к обратной связи»;
- по фидбеку — правки и **следующий батч** из трёх задач, пока план не закрыт.

Если в батче блокер (падает тест, нет зависимости, неясна инструкция) — **остановиться** и спросить, а не додумывать.

## Мини-пример формулировки плана

Задача: добавить валидацию поля X в API.

- План: файлы хендлера, модели, тесты; шаги — красный тест, реализация, зелёный тест, линт, коммит.
- Выполнение: батч 1 — тест + минимальная реализация; проверка `go test ./...`; чекпоинт.

## Итог

**План → выполнение батчами → верификация → коммит → ревью человеком** даёт воспроизводимый процесс и удобные точки остановки без потери скорости на мелких задачах.

## Гранулярность шагов в плане

Скилл **writing-plans** ориентирует на шаги длительностью **примерно 2–5 минут** одного осмысленного действия, например:

- написать падающий тест;
- запустить и убедиться, что он красный;
- внести минимальный код для прохождения;
- прогнать тесты и линтер;
- зафиксировать коммит.

Так исполнителю не нужно угадывать «что имелось в виду под половиной экрана текста» — каждый пункт проверяем.

## Где писать план и контекст

Для крупных изменений разумно вести план в отдельной ветке или **worktree**, чтобы не смешивать черновики плана с несвязанными правками. Точное правило — в **brainstorming** / договорённостях команды; важно, чтобы путь к файлу плана был известен всем, кто продолжает работу.

## Связь с формализацией задачи

Если вход сырой («сделайте удобнее оплату»), перед **writing-plans** полезен **task-formalizer**: вынести AC/DoD, вопросы P0–P2, пробелы. Иначе план будет строиться на домыслах и чаще переписывается после первого чекпоинта.

## Вариант: план как задачи в трекере

Не всегда план остаётся файлом в репозитории. Для крупных рефакторингов, затрагивающих несколько подсистем, план удобнее материализовать **как иерархию задач в трекере** (Kaiten):

- **Родительская карточка** = цель и acceptance criteria всей работы.
- **Подзадачи** = шаги плана, каждый самодостаточен и может выполняться параллельно.
- **Чеклист AC** на родительской = критерии «всё готово».

Преимущества перед файлом плана:

| Файл в репозитории | Задачи в трекере |
|---|---|
| Виден только в ветке/PR | Виден всей команде |
| Статус — вручную | Статус — колонки Kaiten |
| Нет привязки к исполнителям | Можно назначить ответственных |
| Нет истории обсуждения | Комментарии на карточке |

Когда использовать этот вариант:

- Работа займёт больше одной сессии.
- Задачи могут выполняться разными людьми или в разном порядке.
- Нужна видимость прогресса для команды или менеджера.

AI может создать всю иерархию за одну сессию через MCP (Jarvis): `create_card` для каждой задачи, `add_child_card` для связей, `create_checklist` для AC. Пример — [рефакторинг ошибок плеера](../cases/player-error-handling-refactor.md): 1 родительская + 7 подзадач + чеклист из 7 пунктов AC, созданные за 10 минут без выхода из IDE.

Файл плана и задачи в трекере не исключают друг друга: план фиксирует техническую детализацию (архитектура, конкретные файлы, шаги), а карточки — управление работой (статус, ответственные, сроки). Для крупных задач полезны оба.

## Когда упростить цепочку

Для правки опечатки в README или одной очевидной строки полный многостраничный план избыточен. Паттерн «план → выполнение» масштабируйте под риск: чем выше риск регресса и чем больше затронутых файлов, тем строже стоит придерживаться письменного плана и батчей с верификацией.
