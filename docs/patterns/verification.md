# Верификация перед завершением

Документ отражает скилл **verification-before-completion**: любые утверждения о готовности опираются на **свежие** доказательства из запущенных команд, а не на ожидания или прошлые прогоны.

## Принцип

**Evidence before claims, always.** Утверждать успех без проверки в том же сеансе работы — значит подменять факт уверенностью.

## Железное правило

```
NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE
```

Если проверяющая команда **не была запущена** в контексте текущего ответа (или явно не показан её полный актуальный вывод), **нельзя** утверждать, что «всё проходит», «сборка зелёная», «баг исправлен».

## Функция-шлюз

Перед любой формулировкой о статусе:

1. **IDENTIFY** — какая команда **доказывает** это утверждение?
2. **RUN** — выполнить команду **полностью**, заново (не фрагмент, не «почти то же»).
3. **READ** — прочитать весь вывод, код выхода, число ошибок/падений.
4. **VERIFY** — подтверждает ли вывод claim? Если нет — сообщить фактический статус **с цитатой/выводом**.
5. **CLAIM** — только после шага 4 формулировать успех **вместе с указанием, что было запущено и что увидели**.

Пропуск шага = это не верификация.

## Типовые проверки

Подставьте команды вашего репозитория:

| Область | Примеры команд |
|---------|----------------|
| Go | `go build ./...`, `go vet ./...`, `go test ./...` |
| Фронтенд | `npm run lint`, `npm run build`, `npx eslint ...` |
| Монорепозиторий | те же цели через Makefile / task runner, указанный в проекте |

«Линтер чист» не заменяет «компилятор прошёл», если вы заявляете про сборку.

## Анти-паттерн

ИИ (или человек) пишет «готово», «исправлено», «тесты должны проходить», **не запуская** ни одной проверки или ссылаясь на прогон «вчера». Это ровно то, против чего направлен скилл: **сначала доказательство, потом формулировка.**

## Красные флаги в формулировках

- «должно работать», «скорее всего», «похоже, всё ок»;
- радость или закрытие задачи **до** вывода команд;
- доверие к отчёту другого агента без самостоятельного прогона.

## Краткий чек-лист перед коммитом / MR

- Определена минимальная команда верификации для этой задачи.
- Команда выполнена, выход просмотрен.
- В описании коммита/MR при необходимости указано, что проверено (по договорённости команды).

Так снижается риск регрессий и недоверия к автоматизации: завершение работы измеряется артефактом терминала, а не намерением.

## Таблица: что считается доказательством

Ниже — ориентир из скилла; подставьте команды своего CI и Makefile.

| Утверждение | Нужно | Недостаточно |
|-------------|--------|----------------|
| Тесты проходят | Вывод команды тестов: 0 падений | «Вчера проходили», «должны» |
| Линтер чист | Вывод линтера: 0 ошибок | Проверили один файл из десяти |
| Сборка ок | Команда сборки: код выхода 0 | Только линтер без компиляции |
| Баг исправлен | Воспроизведение симптома + проверка фикса | «Код поменял» |
| Регресс-тест есть | Цикл красный → зелёный зафиксирован | Тест прошёл один раз без RED |

## Рационализации, которые не работают

Скилл явно отсекает отговорки вроде «уверен, что сработает», «только в этот раз», «устал — пропущу проверку», «другой агент сказал успех». Для инженерного процесса единственный приемлемый ярлык успеха — **свежий** артефакт команды.

## Связь с merge request

Перед созданием MR имеет смысл прогнать **тот же** набор проверок, что и локально (или дождаться зелёного pipeline), и в описании указать, что именно запускалось. Это снижает цикл «MR открыт — CI красный — возврат на доработку».

## Минимальный стандарт для агента

Если агент сообщает о завершении задачи, ожидаемый формат ответа включает: **какую команду выполнил**, **какой был код выхода / сколько тестов упало**, **краткий вывод**. Без этого сообщение не выполняет железное правило скилла, даже если текст звучит уверенно.
