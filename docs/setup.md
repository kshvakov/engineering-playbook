# Установка и настройка

Краткая инструкция: Cursor, модель, MCP, скилы, правила и проверка, что всё завелось.

## 1. Cursor IDE

1. Скачайте установщик с [cursor.com](https://cursor.com) (Linux / macOS / Windows).
2. Установите и запустите Cursor. Войдите в аккаунт, если IDE попросит (подписка и лимиты завязаны на аккаунт).

Рекомендуется держать Cursor в актуальной версии: настройки MCP и чата периодически меняются.

## 2. Модель AI (Claude — основная)

1. Откройте настройки Cursor (обычно **File → Preferences → Cursor Settings** или аналог в вашей ОС).
2. В разделе моделей выберите **Claude** как основную модель для чата и агента (конкретное имя в интерфейсе может быть вроде Sonnet / Opus — ориентируйтесь на политику компании и доступность в вашем тарифе).
3. При необходимости включите **Long context** или аналог, если работаете с большими репозиториями.

Если Claude в вашей организации недоступен, зафиксируйте в команде запасной вариант и используйте его единообразно.

## 3. MCP-серверы

MCP даёт ассистенту доступ к Kaiten, Пачке, БД и т.д. Подключение настраивается в конфигурации MCP Cursor (JSON). Точный путь к файлу смотрите в актуальной документации Cursor: как правило, это **Cursor Settings → MCP** или отдельный файл конфигурации MCP.

Подключите минимум две интеграции (имена серверов в конфиге могут отличаться — приведите их к тому, что вы реально ставите):

| Назначение | Сервер | Зачем |
|------------|--------|--------|
| Таск-трекер + мессенджер | **Jarvis** | Карточки Kaiten, каналы/треды Пачки, привязка веток к задачам |
| Базы данных | **kinescope-dev-mcp** | Схема, здоровье, разбор запросов PostgreSQL и ClickHouse |

Настройка MCP: откройте **Cursor Settings → MCP** и добавьте серверы. Конфигурация в формате JSON:

```json
{
  "mcpServers": {
    "kinescope-dev-mcp": {
      "url": "http://10.1.2.51:8617/mcp"
    },
    "jarvis": {
      "url": "https://jarvis.kinescope.dev/mcp",
      "type": "streamable-http",
      "headers": {
        "X-Kaiten-Token": "<ваш токен Kaiten>"
      }
    }
  }
}
```

- `kinescope-dev-mcp` работает по HTTP без авторизации (доступен из внутренней сети / VPN).
- `jarvis` требует токен Kaiten в заголовке `X-Kaiten-Token`. Получить токен: Kaiten → Настройки профиля → API-токены.
- Токены не коммитьте в репозиторий.
- После правки конфига перезапустите Cursor или нажмите кнопку перезагрузки MCP в настройках.

Подробности по сценариям Kaiten и БД см. в playbook: [Таск-трекер](mcp/task-tracker.md), [Базы данных](mcp/databases.md).

## 4. Скилы (инструкции для AI)

Скилы лежат в репозитории playbook в каталоге `skills/`. Их нужно скопировать в пользовательскую папку Cursor, чтобы ассистент их подхватывал.

```bash
mkdir -p ~/.cursor/skills
cp -r /path/to/engineering-playbook/skills/* ~/.cursor/skills/
```

Проверка:

```bash
ls ~/.cursor/skills | head
```

Должны появиться каталоги вроде `task-formalizer`, `writing-plans`, `verification-before-completion` и т.д.

Если вы обновляете playbook из Git, повторите копирование или используйте `rsync`, чтобы не затирать локальные правки без необходимости:

```bash
rsync -a --delete /path/to/engineering-playbook/skills/ ~/.cursor/skills/
```

Флаг `--delete` удалит из `~/.cursor/skills/` то, чего больше нет в источнике; используйте только если вы не храните там эксклюзивно свои скилы.

## 5. Пользовательские правила (User Rules)

В Cursor задайте **User Rules** (глобальные правила для чата/агента). Откройте настройки Cursor и найдите поле вроде **Rules for AI** / **User Rules**.

Примеры правил (вставьте и адаптируйте под себя):

**Язык ответов**

```text
В чате пиши на русском языке.
```

**Маршрутизация ролей (если используете единый формат из внутреннего playbook)**

```text
Перед планированием и перед финальным ответом:
1) Выбери base-роль (одна) и при необходимости checker-роли.
2) Выведи маркеры: [ROUTER]: selected skills = …; [TRACE] read: …
3) Дальше в сообщениях используй префиксы ролей: [ROLE]: … и маркеры переключения роли.
```

Дополнительно имеет смысл явно прописать:

- репозитории `gitlab.kinescope.dev`: коммиты и комментарии в коде на русском;
- предпочтение запускать команды самому агенту, а не только выдавать инструкции человеку;
- запрет на лишние рефакторинги вне задачи.

Правила на уровне **проекта** можно дублировать в `.cursor/rules/` — тогда они применяются только в этом репозитории.

## 6. Проверка, что всё работает

Выполните по очереди.

1. **Чат**  
   Откройте проект, задайте вопрос: «Кратко опиши структуру каталогов в корне репозитория». Ответ должен опираться на реальные файлы.

2. **MCP Jarvis**  
   Напишите в чат: «Покажи мои задачи в Kaiten». Если ассистент не может обратиться к Kaiten — проверьте JSON MCP, токен и логи Cursor.

3. **MCP БД**  
   Напишите: «Проверь здоровье PostgreSQL» или «Покажи список таблиц в ClickHouse». Ошибка подключения почти всегда означает неверный URL или сеть/VPN.

4. **Скилы**  
   Напишите: «Формализуй задачу: добавить кнопку X в админку». Если ассистент не подхватывает скил task-formalizer — проверьте путь `~/.cursor/skills/` и перезапустите Cursor.

5. **Сборка (по стеку проекта)**  
   В терминале из корня репозитория:

   ```bash
   go build ./...
   ```

   или

   ```bash
   npm run build
   ```

   Убедитесь, что базовая команда проходит без ваших правок — так вы отделяете проблемы среды от проблем кода.

Когда эти пункты зелёные, можно переходить к [Первый рабочий день с AI](first-day.md).
