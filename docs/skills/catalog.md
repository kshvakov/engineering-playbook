# Каталог скилов

Ниже — скилы, входящие в **engineering-playbook** (каталог `skills/` репозитория). Используйте таблицу как быстрый указатель: **когда** подключать скилл, а не как дословно пересказывать его содержание.

| Скил | Назначение | Когда использовать |
|------|------------|-------------------|
| **using-superpowers** | Мета-скил: перед ответом проверять каталог скилов и подключать релевантные. | Практически всегда в начале сессии и при смене типа задачи. |
| **task-formalizer** | Превращает сырую задачу в пакет артефактов: spec, AC/DoD, вопросы P0–P2, пробелы, тест-матрица. | Новая или размытая задача до планирования и кода. |
| **writing-plans** | Пошаговый план реализации: файлы, шаги, тесты, коммиты; формат «кусочков на 2–5 минут». | После согласования требований, **до** правок в коде. |
| **executing-plans** | Выполнение готового плана батчами (по умолчанию по 3 задачи) с чекпоинтами на фидбек. | Когда есть файл плана и нужно идти по нему в отдельной сессии или итерации. |
| **verification-before-completion** | Запрет утверждений «готово / проходит» без свежего вывода проверяющих команд. | Перед любым заявлением о завершении, перед коммитом/push/MR. |
| **systematic-debugging** | Систематический разбор: симптом → гипотезы → доказательства → root cause до фикса. | Любой баг, флаппинг тест, неочевидный сбой. |
| **brainstorming** | Проработка идей, границ, рисков и вариантов дизайна **до** написания кода. | Новая фича, крупный рефакторинг, неясные требования. |
| **finishing-a-development-branch** | Чеклист завершения ветки: тесты, опции merge/PR, аккуратное закрытие работы. | Когда реализация сделана и нужно безопасно довести до merge. |
| **requesting-code-review** | Организация ревью (в т.ч. через субагента-ревьюера) по готовой фиче или крупному диффу. | После фичи или перед merge, когда нужен внешний взгляд. |
| **dispatching-parallel-agents** | Параллельный запуск агентов на **независимые** задачи без общего изменяемого состояния. | Две и более независимые дорожки (разные файлы, разные причины сбоев). |
| **go-engineer** | Роль и практики Go-разработчика: корректность, надёжность, контракты, observability (пример domain-specific скилла). | Сервисы и библиотеки на Go в продакшене. |

## Ops-скилы

Помимо dev-скилов, в личном `~/.cursor/skills/` доступны скилы для инфраструктурных задач. Они не входят в каталог `skills/` репозитория, но активно используются в ops-цикле ([Ops и инфраструктура](../workflow/ops-infrastructure.md)).

| Скил | Назначение | Когда использовать |
|------|------------|-------------------|
| **linux-engineer** | Linux-диагностика в стиле Brendan Gregg: USE/RED-методология, CPU/память/диск/сеть, безопасные наблюдения, copilot-режим. | Алерты, деградации, «сервер тормозит», bottleneck analysis. |
| **linux-admin** | Безопасные state-changing операции: пользователи, SSH, sudo. Обязательный rollback-план и проверка доступа перед изменениями. | Управление доступом, hardening, SSH-ключи. |
| **bender** | Ops-роль для инфраструктурных MCP-инструментов: Proxmox, мониторинг, инвентаризация. | Управление инфраструктурой через MCP, capacity planning. |

## Как читать «Назначение»

Скилл в Cursor/Claude описывается в **SKILL.md** с полями frontmatter `name` и `description`. Поле `description` в официальном стиле формулируется как **когда** применять скилл (триггер), а не пересказ всего содержания. В таблице выше «Назначение» слегка шире, чтобы человеку было проще ориентироваться.

## Сочетания

Типичные цепочки:

- **using-superpowers** → **task-formalizer** → **writing-plans** → **executing-plans** → **verification-before-completion** → **finishing-a-development-branch** / **requesting-code-review**.
- На багах: **systematic-debugging** на всём пути; верификация — **verification-before-completion**.
- Много независимых поломок: **dispatching-parallel-agents** + затем общая верификация репозитория.
- Ops-задачи: **linux-engineer** (диагностика) или **bender** (MCP/Proxmox) → параллельный SSH → отчёт.

## Где лежат файлы

В репозитории playbook скилы находятся в `skills/<имя-скилла>/SKILL.md`. Локальная копия у разработчика может быть продублирована в `~/.cursor/skills/` (или каталоге агента по документации платформы) — важно не плодить расхождения: источник истины для команды — git в **engineering-playbook**.

## Обновление каталога

При добавлении нового скилла в репозиторий обновите эту таблицу и кратко опишите триггер использования в том же формате. Так README и онбординг останутся согласованными с фактическим содержимым `skills/`.

## Как не перегрузить сессию

Один запрос редко требует одновременного «включения» всех одиннадцати скилов. Практичный порядок:

1. **using-superpowers** — осознанно просмотреть каталог.
2. Один «главный» скилл по типу задачи (план, отладка, параллель, завершение ветки).
3. **verification-before-completion** держать в голове как инвариант перед заявлениями о готовности, даже если отдельный вызов инструмента не требуется.

Перегрузка инструкциями ухудшает соблюдение; лучше меньше релевантных скилов, чем все подряд.

## Domain-specific скилы вне таблицы

В личном `~/.cursor/skills/` у вас могут быть дополнительные скилы (**go-kinescope**, **clickhouse-best-practices**, внутренние runbook'и). Они **не** перечислены в таблице выше: таблица отражает только набор **engineering-playbook**. При публикации нового доменного скилла в общий playbook решайте отдельно: это общекомандной уровень или личная настройка.

## Мини-FAQ

**Нужен ли writing-plans для одной строки?** Обычно нет; достаточно аккуратного коммита и локальной проверки. План масштабируйте под объём и риск.

**executing-plans без файла плана?** Скилл предполагает **готовый** документ плана. Если его нет, сначала **writing-plans** или краткая фиксация шагов в задаче.

**Чем systematic-debugging отличается от verification-before-completion?** Первый — **как** найти причину до фикса; второй — **когда** можно сказать «готово» после изменений. Они дополняют друг друга.

**Можно ли заменить go-engineer на «просто спроси ИИ»?** Можно, но скилл фиксирует продакшен-фокус: контракты, ошибки, наблюдаемость. Для сервисов на Go это снижает разброс стиля между сессиями.

## Версионирование

Текст скилла в git имеет историю: при сомнениях смотрите diff в **engineering-playbook**. Если локальная копия в `~/.cursor/skills/` расходится с репозиторием, приоритет для командной работы — версия из git, затем синхронизация локального каталога.
