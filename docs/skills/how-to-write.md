# Как писать свои скилы

Материал опирается на скилл **writing-skills**: создание скилла — это **TDD для процессной документации**, а не разовый рассказ «как я однажды починил прод».

## Что такое скилл

Скилл — это **справочник по проверенному приёму**: паттерн, чеклист, способ отладки, роль. Он помогает следующему запуску агента найти и применить согласованный подход.

Скилл **не** заменяет:

- разовый постмортем одной инцидент-сессии;
- правила, которые можно полностью автоматизировать проверкой в CI;
- узкие конвенции одного репозитория (часть этого лучше держать в `AGENTS.md` / `CLAUDE.md` репозитория).

## Где хранить личные скилы

Распространённый путь для Cursor:

`~/.cursor/skills/<имя-скилла>/SKILL.md`

В других средах путь может отличаться (например, каталоги Claude Code / Codex — смотрите документацию инструмента). Имя папки совпадает с идентификатором скилла в нижнем регистре и с полем `name` в frontmatter.

Командные скилы разумно вести в репозитории **engineering-playbook** и синхронизировать у себя копированием или submodule — так не теряется версия, согласованная с командой.

## Структура SKILL.md

1. **Frontmatter (YAML)** — поддерживаются только два поля: `name` и `description`.
   - `name`: латиница, цифры, дефисы.
   - `description`: **третье лицо**, формулировка **когда** использовать («Use when…» / «Используйте, когда…»), без пересказа всего содержимого; лимит символов — по правилам платформы (ориентир до ~1024 символов суммарно с name).

2. **Тело** — принципы, шаги, таблицы, примеры команд, анти-паттерны. Длинные справочники (сотни строк API) выносят в отдельные файлы.

## Каталог `references/` и соседние файлы

Если в скилле нужны:

- большие шаблоны вывода;
- подробные чеклисты;
- вспомогательные сценарии;

их кладут рядом, например:

```
~/.cursor/skills/go-kinescope/
  SKILL.md
  references/
    patterns.md
    checklists/
      deploy.md
```

В **SKILL.md** остаётся «навигация и правила», а тяжёлый текст — в `references/`, на который скилл ссылается.

## TDD-подход к написанию

Цикл из **writing-skills** / **test-driven-development**:

1. Сформулировать **давление-сценарий** (например, субагент без скилла получает задачу, где легко нарушить правило).
2. Прогнать **базовую линию**: зафиксировать, как агент **ошибается** (рационализации, пропуск шагов).
3. Написать **минимальный** SKILL.md, закрывающий именно эти нарушения.
4. Повторить сценарий: агент со скиллом должен **соблюдать** правила.
5. **Рефакторинг** текста скилла: закрыть новые лазейки, не ломая прохождение сценария.

Если вы не видели провала без скилла, вы не знаете, учит ли скилл нужному.

## Жёсткие и гибкие скилы

Удобная практическая градация:

- **Жёсткий (rigid):** пошаговый алгоритм, мало места для толкования — отладка, верификация перед «готово», выполнение плана батчами. Нарушение шага считается ошибкой.
- **Гибкий (flexible):** принципы и критерии выбора — архитектурные роли, мозговой штурм, код-ревью. Агент адаптирует детали под контекст, но сохраняет инварианты (например, «сначала root cause»).

В терминах **writing-skills** рядом стоят типы **technique** (конкретная процедура), **pattern** (способ мыслить), **reference** (справочник). Жёсткость чаще у technique, гибкость — у pattern.

## Пример: доменный скилл вроде go-kinescope

1. Выделить повторяющиеся решения вашей организации: логирование, ошибки, gRPC, миграции.
2. Оформить **go-kinescope** как скилл с `description`: «Используйте для сервисов Kinescope на Go…».
3. В `references/` положить шаблоны ответов инциденту, чеклисты ревью, ссылки на внутренние стандарты.
4. Прогнать 2–3 реальных задачи и дописать раздел «анти-паттерны», если агент всё ещё находит лазейки.

## Перед «релизом» скилла

- Прочитать скилл свежим взглядом: понятен ли триггер из одного только `description`?
- Убедиться, что нет дублирования с существующими скиллами без отсылки «когда выбрать A, а когда B».
- Один скилл — одна тема; «всё о компании в одном файле» разъедется и перестанет вызываться целенаправленно.

Хороший скилл коротко говорит **когда** его открыть и достаточно подробно говорит **что делать** после открытия.
