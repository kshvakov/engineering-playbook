# Мультирепозиторная работа

Многие фичи затрагивают **несколько репозиториев** сразу: контракт API и UI, миграции и отображение, общие соглашения об ошибках. С ИИ в Cursor удобнее всего держать **одинаковое имя ветки** и одинаковый суффикс Kaiten во всех клонах, чтобы не путать контекст в чате, в MR и в трекере.

## Пример из практики: botdetect в суперадминке

Задача: админ‑раздел **botdetect** (подозрительные IP, правила, работа с аналитикой).

Типичная пара репозиториев:

- **Backend (Go):** `kinescope-superadmin` — HTTP/gRPC‑хендлеры, авторизация, запросы в **ClickHouse**, миграции **PostgreSQL** для справочников, настроек или учётных таблиц, если они живут не в аналитике.
- **Frontend (React):** `kinescope-superadmin-frontend` — маршруты, таблицы и формы на **Ant Design**, вызовы API, состояние фильтров, обработка загрузки и ошибок.

Имя ветки, согласованное с карточкой Kaiten:

`feature/botdetect-suspicious-ip-kaiten-62353324`

Такое же имя в обоих клонах упрощает поиск веток, ссылки в описании MR и обсуждение с ИИ: достаточно назвать одну строку.

## Старт работы в каждом репозитории

В каждом клоне:

```bash
git checkout master   # или main — как принято в проекте
git pull
git checkout -b feature/botdetect-suspicious-ip-kaiten-62353324
```

Если ветка уже существует на origin после чужого push, используйте `git fetch` и `git checkout` по удалённому имени. Не смешивайте в одном клоне правки двух репозиториев — у ИИ путаются импорты и пути.

## Порядок зависимостей

Обычно сначала стабилизируют **контракт API** (поля, коды ошибок, пагинация), затем подключают UI. На практике возможны параллельные MR, если фронт временно мокает ответ или использует feature flag, но договорённость «сначала бэкенд» снижает количество переделок в React.

Для botdetect типичная цепочка: миграции PostgreSQL (если нужны) → хендлер и запрос к ClickHouse → контракт в OpenAPI/доке → экран во фронтенде.

## Параллельные изменения

- **Backend:** новые эндпоинты или поля ответа, валидация query/body, таймауты и лимиты на CH, единый формат ошибок.
- **Frontend:** компоненты таблицы, фильтры, синхронизация с URL, доступ к разделу только роли админа.

План (**writing-plans**) ведите в одном репозитории как «источник правды» или разделите явно: файл в backend с секцией Frontend и ссылками на пути во втором репо. Главное — в каждом шаге указать **полный путь репозитория** в тексте задачи, чтобы исполнитель не открыл не тот root в Cursor.

## Параллельная верификация

Перед коммитом в соответствующем дереве:

- в Go: `go build ./...`, `go vet ./...`, целевые тесты;
- во фронтенде: `eslint` и при необходимости `npm run build` или проверка типов.

Два терминала или два окна Cursor с разными корнями — нормальная практика. ИИ может выполнять команды по очереди; вы можете явно сказать: «сейчас workspace — frontend».

## Параллельный push и MR

После локальных коммитов в каждом репозитории:

```bash
git push -u origin feature/botdetect-suspicious-ip-kaiten-62353324
```

В описании каждого MR укажите:

- ссылку на карточку Kaiten;
- ссылку на «братский» MR, если без него фича не работает end‑to‑end;
- кратко, какой MR должен мержиться первым.

Так ревьюеры видят полную картину и не мержат UI раньше API или наоборот. Если в команде принят чек‑лист «оба MR зелёные в CI», добавьте его в шаблон описания.

## Работа с ИИ в чате

- Один чат на один корень репозитория — меньше галлюцинаций путей.
- Для сквозных решений (JSON ответа) сначала зафиксируйте пример payload в backend‑ветке или в задаче, затем вставьте его во фронтенд‑сессию.
- При использовании MCP к БД проверяйте запросы в том окружении, которое соответствует ветке (dev/stage), и не смешивайте креды в логах чата.

## Итог

Мультирепозиторность не отменяет цепочку **формализация → план → пакетное выполнение**; она требует **одинаковых имён веток**, явной привязки шагов плана к репозиторию и явных ссылок между MR. Тогда Cursor и ИИ остаются помощниками, а не источником путаницы между Go и React.
