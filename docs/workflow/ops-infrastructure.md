# Цикл ops/инфраструктурных задач с AI в Cursor

Цикл разработки (`task-formalizer` → `writing-plans` → `executing-plans`) заточен под код: формализация, файлы, коммиты, MR. Ops-задачи устроены иначе: алерт нужно разобрать здесь и сейчас, формализация избыточна, а «план» — это последовательность диагностических команд, а не шаги с файлами. Этот документ описывает рабочий цикл для таких задач.

## Типичные триггеры

- Алерт из мониторинга: disk full, RAID degraded, high CPU/memory.
- Запрос на инвентаризацию: список VM, модели дисков, состояние RAID.
- Capacity planning: где мало ресурсов, что под замену.
- Troubleshooting: сервис не отвечает, высокий latency.

## Цикл ops-работы

```
Алерт / запрос
    ↓
Роутинг на скилл (linux-engineer / bender)
    ↓
Сбор данных (SSH / MCP)
    ↓
Анализ и гипотезы
    ↓
Действие (copilot-режим: предлагает, не выполняет)
    ↓
Отчёт / артефакт
```

Роутинг определяет скилл по типу задачи. Для диагностики хоста — `linux-engineer`, для инвентаризации через Proxmox — `bender`, для изменений в доступе — `linux-admin`. Дальше — сбор данных, анализ, действие, фиксация результата.

## Инструменты

**SSH из Cursor** — прямое подключение к серверам. AI строит и выполняет read-only команды (`df`, `du`, `lsof`, `smartctl`, `mdadm`, `cat /proc/mdstat`, `ss`, `ip`). Опасные действия (`kill`, `mount`, `systemctl restart`) — только с подтверждением.

**MCP (`kinescope-ops-loc`)** — Proxmox API: `pve_list_vms`, `pve_list_hosts`. Инвентаризация без SSH, агрегация данных, capacity alerts.

## Скиллы

| Скилл | Роль в ops |
|-------|-----------|
| `linux-engineer` | Диагностика по USE/RED: CPU, память, диск, сеть. Copilot-режим по умолчанию. |
| `linux-admin` | State-changing ops: пользователи, SSH, sudo. Обязательный rollback-план. |
| `bender` | Инфраструктурные MCP-инструменты: Proxmox, мониторинг. |

## Copilot-режим

В разработке AI свободно редактирует код. В ops по умолчанию **copilot**: AI предлагает команды и план, state-changing действия — только после явного подтверждения оператора.

Безопасные (AI выполняет сам):
`df`, `du`, `lsof`, `top`, `iostat`, `ss`, `cat /proc/*`, `smartctl -i/-H`, `mdadm --detail`, `lsblk`.

Требуют подтверждения:
`kill`, `systemctl restart`, `mount`, `sysctl -w`, `mdadm --add/--remove`, любые записи.

## Параллельный сбор

Когда нужно проверить несколько серверов — AI запускает SSH-команды параллельно (несколько tool calls в одном ответе). Результаты собирает в единый отчёт. Это критично для инвентаризации: обход 10+ хостов последовательно занимает минуты, параллельно — секунды.

## Артефакты

После диагностики AI формирует структурированный вывод: таблицы, списки, рекомендации. При необходимости сохраняет в файл (`low_cap.md`, `raid-status.md`) для дальнейшего использования или передачи коллегам.

## Связь с dev-циклом

Ops-задачи иногда переходят в dev: найденный баг (зависший ffmpeg, утечка дескрипторов) → задача в Kaiten → формализация → фикс. Тот же Cursor, те же MCP, но переключение с `linux-engineer` на `go-engineer` / `systematic-debugging`. Полный dev-цикл описан в [overview.md](overview.md).

## Типичные ошибки

- Выполнять опасные команды без copilot-режима.
- Диагностировать «наугад» без USE-метода.
- Не фиксировать результат диагностики (ни в чате, ни в файле).
- Игнорировать MCP и делать всё через SSH, когда есть готовые инструменты.
- Не проверять rollback-план перед state-changing операцией.
