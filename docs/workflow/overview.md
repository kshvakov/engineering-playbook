# Полный цикл работы с ИИ в Cursor

Этот документ описывает сквозной цикл: от формулировки задачи до итераций после ревью. Он опирается на реальный опыт: админ‑раздел для **botdetect** (подозрительные IP, правила, отчёты) с одновременной работой в бэкенде на **Go**, фронтенде на **React**, запросами в **ClickHouse**, миграциями **PostgreSQL** и UI на **Ant Design**. Параллельно используются **MCP** (Kaiten, БД, при необходимости GitLens) и **скиллы** из engineering-playbook.

## Зачем фиксировать цикл

Без явных шагов ИИ быстро уходит в реализацию «как кажется правильным»: путаются источники данных, пропускаются AC, коммиты раздуваются. Цикл ниже связывает инструменты Cursor с дисциплиной проверки и трекинга задач.

## 1. Получение задачи

Источник может быть любым:

- карточка **Kaiten** — просто попросите ассистента показать задачу, и он подтянет заголовок, описание, ссылки на GitLab, тред в мессенджере;
- вставка текста в чат Cursor;
- устная формулировка, записанная в чат как есть.

Сохраните **исходный текст** и ссылки: они пойдут во вход формализатора и в описание MR. Если MCP подтянул карточку, приложите краткое резюме для чата, чтобы контекст не потерялся между сессиями.

## 2. Формализация

Подключается скилл **task-formalizer**: из «сырого» текста получается **Spec v1** с критериями приёмки (AC), определением готовности (DoD), вопросами по приоритетам P0–P2, явными пробелами и тестовой матрицей. Если не хватает данных, скилл останавливается по **stop conditions** и не заполняет выдуманными деталями.

Для botdetect это переход от «нужен раздел в админке с подозрительными IP» к списку экранов, полей таблицы, источников данных (ClickHouse vs PostgreSQL), прав доступа и сценариев проверки. Пока P0 не закрыты, планирование кода лучше не начинать.

## 3. Планирование

После утверждённого Spec подключается **writing-plans**: пошаговый план с мелкими шагами (ориентир **2–5 минут** на шаг), путями файлов, фрагментами кода или SQL, командами проверки. План сохраняется в репозитории, например `docs/plans/YYYY-MM-DD-botdetect-admin.md`, чтобы его можно было выполнять в отдельной сессии или передать коллеге.

## 4. Создание задачи и веток

- В Kaiten создаётся или обновляется карточка — просто попросите ассистента «создай задачу и привяжи ветку», и он сам обратится к Kaiten через MCP.
- Ветка в Git: единый стиль с суффиксом карточки, например `feature/botdetect-suspicious-ip-kaiten-62353324`.

Если затронуты два репозитория (**kinescope-superadmin** и **kinescope-superadmin-frontend**), **одно и то же имя ветки** заводится в обоих — проще MR, ссылки и обсуждение с ИИ.

## 5. Выполнение

Скилл **executing-plans** загружает план, критически его просматривает, заводит todo и выполняет задачи **пакетами** (по умолчанию **первые три задачи**), затем останавливается на отчёте «готово к обратной связи». Так снижается риск уехать далеко в неверном направлении.

Пример пакетов для botdetect: первый — миграция схемы `botdetect`, HTTP‑хендлер, обвязка запроса к ClickHouse; второй — страница React, таблица Ant Design, вызов API и обработка ошибок.

## 6. Верификация

Перед каждым коммитом прогоняйте минимальный набор для затронутых частей:

- **Go:** `go build ./...`, `go vet ./...`, при наличии — тесты затронутого пакета;
- **фронтенд:** `eslint` и typecheck, если он обязателен в проекте.

Скилл **verification-before-completion** в том же духе: не утверждать «готово», пока команды реально не выполнены и вывод не просмотрен. MCP к ClickHouse/PostgreSQL помогает проверить запросы на стенде, но не заменяет локальные команды сборки.

## 7. Коммит и push

Сообщения коммитов и комментарии в коде для репозиториев Kinescope — **на русском**. Дробите коммиты по плану (миграция, API, UI), чтобы ревьюеру было проще пройтись по истории. После `git push` открывайте MR с ссылкой на Kaiten и, при мультирепо, на связанный MR.

## 8. Итерация

Вы проверяете поведение локально или на стенде, оставляете замечания; ИИ вносит правки тем же циклом (план при необходимости уточняется, новые шаги выполняются пакетами). Перед слиянием подключайте **requesting-code-review** для отдельного прохода по диффу.

## Сводка: скиллы и инструменты

| Этап            | Скилл / инструмент |
|-----------------|--------------------|
| Задача          | чат (Kaiten через MCP) |
| Формализация    | task-formalizer    |
| План            | writing-plans      |
| Выполнение      | executing-plans    |
| Проверка        | verification-before-completion, go/eslint |
| Завершение ветки| finishing-a-development-branch |
| Ревью           | requesting-code-review |

**Краткая связка:** task-formalizer → writing-plans → executing-plans → verification → finishing / requesting-code-review. MCP сокращает ручной копипаст; скиллы задают, **когда** остановиться и **что** считать доказательством готовности.

## Продолжение работы между сессиями

Крупные задачи редко укладываются в одну сессию с AI. Контекст теряется при закрытии чата. Чтобы следующая сессия подхватила работу без потерь:

1. **Задачи в трекере — основной носитель контекста.** Kaiten-карточки хранят описание проблемы, принятые решения, ключевые файлы. AI в новой сессии может прочитать карточку через MCP и восстановить контекст.

2. **Summary предыдущей сессии.** Cursor автоматически сохраняет summary диалога. При продолжении работы AI видит, что было сделано, какие решения приняты, какие задачи созданы.

3. **Дизайн-документы и планы в репозитории.** Если в сессии создан план или дизайн — он сохранён в файле и доступен в следующей сессии без MCP.

Пример из практики: [рефакторинг ошибок плеера](../cases/player-error-handling-refactor.md) прошёл в две фазы в разных сессиях. Фаза 1 (аудит → задачи) завершилась созданием эпика в Kaiten. Фаза 2 (дизайн UI и телеметрии) началась в новой сессии — AI восстановил контекст из summary и карточек Kaiten, продолжил работу и создал под-эпик внутри существующего эпика.

Рекомендация: завершая сессию, убедитесь, что все принятые решения зафиксированы в карточках или файлах, а не только в тексте чата.

## Типичные ошибки цикла

- **Пропуск формализации:** сразу писать код по сообщению из чата — потом выясняется, что «аналитика» должна была быть в другой витрине ClickHouse.
- **План без верификаций:** шаги «сделать» без команд `go test` / `eslint` — ИИ отмечает задачу выполненной без доказательств.
- **Один огромный коммит:** смешаны миграция, API и UI — откат и ревью превращаются в боль; дробите по плану.
- **Разные имена веток в репозиториях:** путаются MR и ссылки из Kaiten; держите суффикс `kaiten-{id}` везде одинаково.

## Где лежит playbook

Скиллы, на которые ссылается этот цикл, находятся в каталоге `skills/` репозитория **engineering-playbook**. Подключайте их в Cursor через механизм skills вашей среды, чтобы формулировки в документах и поведение ассистента совпадали.
