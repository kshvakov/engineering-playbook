# Планирование и выполнение

Два скилла стыкуются в осознанный цикл: **writing-plans** готовит дорожную карту, **executing-plans** исполняет её с контрольными точками. Оба рассчитаны на Cursor и на то, что исполнитель может не знать ваш репозиторий. Разделение ролей важно: план пишется, когда вы ещё не увязли в диффе; выполнение идёт короткими итерациями с вашим фидбеком.

## writing-plans: план до кода

**Когда:** есть утверждённый spec (например, из **task-formalizer**) или чёткие требования, вы ещё не правите продакшен‑код вслепую.

**Что получается:**

- план с **мелкими шагами** (ориентир **2–5 минут** на шаг): один тест, один проход линтера, один осмысленный коммит — отдельные пункты;
- для каждой задачи — **файлы** (создать / изменить с указанием области строк), **команды тестов**, **ожидаемый вывод** (PASS/FAIL с типичным текстом ошибки);
- при необходимости — фрагменты кода, SQL к ClickHouse или PostgreSQL, конфиги.

Заголовок плана в скилле стандартизирован: цель одной фразой, архитектура в двух-трёх предложениях, стек. В шапке указывается, что для реализации нужен **executing-plans**. План должен быть **DRY**: не дублировать десять раз одну и ту же команду, но и не сваливать всё в «сделать фичу».

**Где хранить:** `docs/plans/YYYY-MM-DD-<feature-name>.md` в репозитории или согласованное с командой место. Имя с датой упрощает поиск и параллельные фичи.

## executing-plans: исполнение с паузами

**Когда:** план записан в файл, вы готовы идти по нему в новой сессии или после перерыва.

**Процесс по скиллу:**

1. **Загрузить и критически прочитать план.** Если есть дыры, невыполнимые шаги или противоречия с репозиторием — обсудить с человеком **до** старта.
2. **Создать todo** по задачам плана (в Cursor — TodoWrite).
3. **Выполнить пакет** — по умолчанию **первые три задачи**: для каждой — `in_progress` → шаги из плана → проверки из плана → `completed`.
4. **Отчёт:** что сделано, вывод верификации, явная пауза для фидбека («готово к обратной связи»).
5. По фидбеку — правки, следующий пакет, пока план не исчерпан.
6. После завершения — **finishing-a-development-branch**: финальные тесты, варианты merge/rebase, решение по MR.

Если план блокирует работу (нет зависимости, падает проверка, формулировка неясна) — **стоп**, уточнение важнее «протиснуться». Скилл прямо запрещает гадать при повторных провалах верификации.

## Как скиллы усиливают друг друга

- План без формализации часто угадывает детали; **task-formalizer** до плана снижает этот риск.
- Пакетное выполнение не даёт ИИ уйти на десять файлов без вашей проверки.
- Мелкие шаги упрощают откат, `git bisect` и объяснение ревьюеру, что зачем менялось.

## Пример: фильтры в таблице подозрительных IP

В админке botdetect уже есть таблица IP из ClickHouse; нужно добавить фильтры по диапазону дат и коду страны.

**Фрагмент плана (writing-plans):**

- **Задача A:** расширить контракт API — query‑параметры `from`, `to`, `country`; валидация в Go; тест на 400 при неверном формате даты.
- **Задача B:** изменить запрос к ClickHouse — условия в `WHERE`, сохранить поведение без фильтров; `go test` на пакет хендлера или интеграционный тест по конвенции репо.
- **Задача C:** React — `DatePicker.RangePicker` и `Select` стран в Ant Design, синхронизация с URL или state, вызов API; `eslint` и ручная проверка в браузере.

**executing-plans** обрабатывает **задачи** плана пакетами по три. Если A, B и C оформлены как три отдельные задачи с заголовками `### Task N`, они попадут в первый пакет; если объединены в одну большую задачу, имеет смысл разнести её в плане на три `Task`, иначе пакет из «трёх задач» будет слишком тяжёлым.

Следующие пакеты могут включать пагинацию с учётом фильтров, права на экспорт, метрики и логирование ошибок CH.

## Практические советы

- В плане явно указывайте **команды** (`go test -run ...`, `npm run lint`), чтобы ИИ не пропустил проверку.
- Для двух репозиториев используйте префиксы `Backend:` / `Frontend:` в названиях задач или два связанных файла плана.
- Не начинайте выполнение с **main/master** без явного согласия пользователя — это требование скилла executing-plans.
- Если в плане указан другой скилл (TDD, debugging), не перескакивайте через него ради скорости.

## Когда пересмотреть план

Возвращайтесь к шагу «загрузка и ревью плана», если изменился контракт API, продукт переопределил AC или выяснилось, что источник данных не ClickHouse, а материализованное представление в PostgreSQL. Патчить код без обновления плана можно только для мелочей, иначе расходится документация и реальность.

## Работа в Cursor на практике

- Откройте root репозитория, к которому относится план, чтобы пути в документе совпадали с файловой системой.
- В начале сессии выполнения вставьте в чат путь к файлу плана и фразу в духе: «используй executing-plans, первый пакет — три задачи».
- После отчёта по пакету просмотрите дифф сами; ИИ склонен к «всё зелёное», пока вы не запустили команды из плана руками или не подтвердили вывод.

## Шаблон задачи в плане

Каждая задача в `writing-plans` хорошо читается, когда содержит:

1. **Цель** одной строкой.
2. **Файлы** с точными путями.
3. **Шаги** нумерованные, каждый с действием и ожидаемым результатом.
4. **Верификация** — команда и что считать успехом.

Так executing-plans не интерпретирует свободный текст и реже ломает порядок шагов.

## Согласование с TDD

Если в команде принят TDD, в плане отдельными шагами идут: написать падающий тест → запустить → реализовать минимум → снова запустить. Скилл writing-plans изначально заточен под такую гранулярность; не сжимайте эти четыре пункта в один «написать тесты и код».

## Оценка длительности

Пакет из трёх задач по 2–5 минут каждая — это ориентир 10–20 минут чистого времени плюс прогон CI. Если задача в плане раздулась до нескольких часов, её стоит разрезать до следующего пакета, иначе контрольная точка с человеком теряет смысл.
