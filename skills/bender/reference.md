## Reference (Brendan Gregg / principal style)
Этот файл — подробные плейбуки и “глубокие” инструменты. Используй их после базовой локализации (RED/USE) и только с учётом ограничений по безопасности/импакту.

### Правило эскалации (обязательно)
Перед `perf`/`eBPF`/`tcpdump` фиксируй: **цель**, **ожидаемый сигнал**, **окно**, **лимиты**, **импакт**, **ИБ (PII/секреты)**, **хранение/чистка**, **явное подтверждение оператора**.

## CPU profiling (Ubuntu/Debian)
### Быстрый чек
- `pidstat -u 1` — кто грузит CPU, user/sys/steal.
- `sar -u 1` — общая картина по CPU.

### perf (только короткими окнами)
Цели: понять, что именно “жрёт” CPU (функции/стек), или оценить CPI/branches/cache-misses.

- **Лёгкая статистика** (обычно безопаснее):
  - `perf stat -p <pid> -- sleep 10`
  - `perf stat -- sleep 10`

- **Горячие функции** (коротко):
  - `perf top -p <pid>` (включать на короткое время, затем остановить)

- **Сэмплинг с записью** (строго лимитировать):
  - `perf record -F 99 -p <pid> -- sleep 15`
  - затем `perf report`

Примечания:
- На проде избегай длинных `perf record`.
- Если нет прав/ядро ограничивает perf — не ломать ограничения; предложить альтернативу (метрики, логи, более узкое окно).

## Off-CPU / latency investigation (eBPF)
Цели: увидеть задержки, которые не объясняются CPU utilization (I/O wait, run queue, scheduler delays).

Подход:
1) Убедись, что базовая картина собрана (RED/USE).
2) Формулируй конкретный вопрос: “где latency?” и “какой тип latency?”.
3) Выбери минимально инвазивный eBPF-инструмент и короткое окно.

Примеры задач (концептуально):
- **Block I/O latency**: выявить медленные операции на блочном уровне и их распределение.
- **TCP retransmits**: подтвердить, что ретраи есть и как часто.
- **Run queue latency / scheduler**: понять, сколько времени треды ждут CPU.

Ограничения:
- eBPF может требовать пакеты (bpftrace/BCC) и права. Если пакетов нет — сначала предложить **обоснование** установки и оценку риска (или альтернативу без установки).
- Всегда задавай лимит по времени (например 30–120 секунд) и объёму вывода.

## Network: tcpdump/pcap безопасно
### Когда уместно
- Есть сетевые симптомы (timeouts/packet loss/retransmits) и базовые счётчики/`ss` не дают ответа.
- Нужно подтвердить конкретный сценарий (RST, handshake, DNS, TLS, MTU).

### Минимальные правила безопасности
- **Фильтр обязателен** (BPF): по хосту/порту/протоколу, избегать “всё подряд”.
- **Лимиты**: время, размер файлов, ring buffer.
- **PII/секреты**: pcap потенциально содержит чувствительные данные. Не выкладывать в публичные места.
- **Хранение**: заранее определить куда кладём, кто имеет доступ, сколько храним, как удаляем.

### Базовые шаблоны (минимально инвазивные)
- Снять “заголовки” без payload:
  - `tcpdump -i <iface> -s 96 -w capture.pcap '<bpf>'`
- Ограничить по времени (примерно):
  - `timeout 30s tcpdump ...`
- Ограничить по размеру/ротации:
  - `tcpdump -C 50 -W 5 -w capture.pcap ...`

Примеры BPF (адаптируй):
- `host 10.0.0.10 and tcp and port 443`
- `udp and port 53`

## Storage/IO: типовые паттерны (в помощь интерпретации)
### Что искать
- **Latency**: рост `await`/`r_await`/`w_await`, хвосты.
- **Saturation**: рост `aqu-sz`/очередей, `%util`.
- **Errors**: I/O error, timeout, reset в `dmesg`/`journalctl -k`.

### Частые причины
- **Writeback pressure**: грязные страницы, фоновые сбросы, заторы на флеше/RAID.
- **FS/journal**: ext4/jbd2, блокировки, sync writes.
- **Device timeouts**: NVMe/SATA resets, ошибки контроллера, деградация массива.

### Safe-first команды
- `iostat -xz 1`
- `pidstat -d 1`
- `journalctl -k -S -1h`
- `lsblk -f`, `df -hT`, `mount`
