# Task Formalizer — playbook: от сырой задачи до релиза

Этот документ — **пошаговый сценарий** для команды: как взять сырую задачу, вычистить противоречия/пробелы, собрать уточнения, обновить исходник и получить артефакты для разработки, дизайна, QA и коммуникаций.

## Оглавление

- 0) Подготовить входную задачу (`task.md`)
- 1) Быстрая классификация (Routing-only)
- 2) Валидация: пробелы и противоречия (без решений)
- 3) Собрать уточнения и входные данные (P0–P2)
- 4) Обновить задачу (v2) и зафиксировать ответы
- 5) Сформировать пакеты по ролям
- 6) Приёмка и релиз: чеклист + release notes
- Примеры end-to-end (2 штуки)
- Шпаргалка ссылок (где что лежит в `references/`)

---

## 0) Подготовить входную задачу (`task.md`)

### Зачем
Снизить шанс домыслов: агент должен опираться на текст и артефакты, а не “угадывать”.

### Вход (что вы готовите)
Файл `task.md` (или секция в таске), минимально:

```markdown
## Контекст
Что происходит и зачем это нужно.

## Цель
Что считаем успехом (наблюдаемо).

## Scope / Out of scope
- In scope: ...
- Out of scope: ...

## Артефакты
- Ссылки/скрины/логи/метрики:

## Ограничения
- Дедлайн/релиз:
- Ограничения/риски, если известны:

## Unknowns (если вы их уже видите)
- ...
```

### Опционально (если вы знаете заранее)

```text
TaskTypeOverride: <UX|Perf|Incident|Migration|SecurityAuthZ|SecurityKeys|Integration>
FormalizerMode: <full|questions_only|routing_only>
```

## 1) Быстрая классификация (Routing-only)

### Цель
Понять, **какой это тип задачи**, какие входы являются блокерами, и что делать дальше.

### Что нового в результате
Сразу после `## Routing` агент возвращает `## Task summary (rewritten)` — переписанное резюме задачи в стиле user-story, строго по фактам входа. Его удобно вставлять в тикет как “читаемую постановку”.

### Промпт (копировать в чат)

```text
Приложи (перетащи) файл с задачей `task.md` в чат и отправь запрос ниже.

Сделай только секцию ## Routing.
Формат секции:
- TaskType
- MustHaveInputs
- RecommendedPrompts
- StopConditions
- RoutingDecision

Запрет:
- не генерируй Spec/AC/TestPack (даже если кажется, что всё понятно)
- не додумывай факты

Если не можешь выбрать тип без домыслов — поставь TaskType=Other и перечисли 2–4 кандидата.
```

### Ожидаемый результат (скелет)

```text
## Routing
TaskType: ...
MustHaveInputs:
- ...
RecommendedPrompts:
- ...
StopConditions:
- ...
RoutingDecision:
- ...
```

### Что делать дальше
- Если `MustHaveInputs` содержит блокеры — переходи к шагу 3 (сбор уточнений).
- Если блокеров нет — всё равно стоит сделать шаг 2 (проверка пробелов/противоречий), чтобы не “закопаться” позже.

---

## 2) Валидация: пробелы и противоречия (без решений)

### Цель
Поймать “разъезжающиеся” требования и критические пробелы **до** того, как вы начнёте писать spec/AC/test.

### Промпт (копировать в чат)

```text
Сделай анализ входной задачи на:
1) пробелы/нехватку входных данных,
2) противоречия/несовместимые требования.

Выведи 2 таблицы:
- Gaps: | ID | Пробел/противоречие | Почему важно | Что уточнить/решить | Приоритет |
- Contradictions: | ID | Противоречие (цитаты/фрагменты) | Почему конфликт | Какой вопрос стейкхолдеру |

Запрет: не предлагай решений и не проектируй реализацию.
```

### Ожидаемый результат
- Таблица пробелов (почему важно, что уточнить, приоритет).
- Таблица противоречий (цитаты, почему конфликт, вопрос стейкхолдеру).

---

## 3) Собрать уточнения и входные данные (P0–P2)

### Цель
Сформировать **короткий список вопросов** и список того, что нужно предоставить (артефакты/данные).

### Промпт (копировать в чат)

```text
Сделай Routing, затем верни только вопросы/Unknowns (P0/P1/P2) и секцию "What to provide next".
Запрет: не генерируй Spec/AC/TestPack/тесты.
Каждый вопрос должен приводить к конкретному решению/данным.
```

### Ожидаемый результат
- `## P0 questions (blockers)` или `## Unknowns (P0 сверху)`
- `## What to provide next`

### Рекомендация по коммуникации
Если нужно отправить вопросы автору задачи/стейкхолдеру — переформулируй P0/P1/P2 в короткое сообщение (8–12 вопросов максимум), добавив к каждому “почему важно” и “что сломается без ответа”.

---

## 4) Обновить задачу (v2) и зафиксировать ответы

### Цель
Свести ответы обратно в исходник, чтобы:
- команда работала “от одного источника правды”;
- изменения были трассируемы.

### Как фиксировать (best practice)
- Сделайте `task.v2.md` (или раздел `## Updates (v2)` внутри `task.md`).
- Добавьте блок “Ответы/решения” с ссылкой на исходные вопросы.

Пример:

```markdown
## Clarifying questions
### P0
- Q-1: ...

## Ответы/решения
- A-1 (на Q-1): ... (кто подтвердил, дата)
```

## 5) Сформировать пакеты по ролям

### Цель
Получить артефакты, которые удобно “раздать” по ролям (dev/design/frontend/QA/marketing) без потери контекста.

### Промпт (full pack)

```text
Собери полный “output pack” одним документом, секциями в таком порядке:
## Routing
## Intake
## Facts / Entities / Invariants
## Gaps / Contradictions / Risks
## Questions (P0/P1/P2)
## Spec v1
## Acceptance Criteria + DoD
## Corner cases
## Test pack
## Docs notes
## Validators (9.1/9.2/9.3)
## ExecutionTrace

Используй только факты из входа и ответы из task.v2.md.
Если чего-то не хватает — не выдумывай; вынеси в Unknowns/Questions/ASSUMPTION.
После сборки прогони валидаторы 9.1–9.3.
```

### Ожидаемый результат
Один “пакет” секциями (Routing → Intake → Facts → Gaps → Questions → Spec → AC/DoD → Test → Docs → Validators → ExecutionTrace).

### Как раздать результат по ролям (что именно копировать)

Практика: держать **один общий пакет** и делать “вырезки” для ролей.

- **Разработчику (dev)**:
  - `Spec v1 (PRD-lite)` + `Acceptance Criteria` + `DoD`
  - `Risk register` (если есть)
  - `Open questions` + `Assumptions` (если остались)

- **Дизайнеру (design / content)**:
  - Для `TaskType=UX`: результат `8.4 UX intake` (baseline/target/inventory/texts) + требования к статусам/ошибкам (если они во входе)
  - `Unknowns` по контенту/экранам/доступности
  - Список решений, которые нельзя принять без стейкхолдера (как вопросы)

- **Фронтенду (frontend)**:
  - “Контракт UI”: статусы, ошибки, empty states, тексты, idempotent/повторяемость действий (если есть)
  - Минимум: `AC` + “как проверить” + список edge/corner кейсов для интерфейса
  - Если информации нет — фронтенд‑часть оформляется как вопросы, а не как “придумаем”

- **QA / тестировщику**:
  - `Test cases` + `Test plan` + `QA notes`
  - Привязка: к каждому `AC` должен быть хотя бы один тест‑кейс

- **Маркетингу / релиз-коммуникациям (marketing)**:
  - `Release notes (user-facing)` + “Кому полезно” + “Как пользоваться”
  - `FAQ` (если появляются повторяющиеся вопросы)

## 6) Приёмка и релиз: чеклист + release notes

### Цель
Чтобы релиз не превращался в “угадайку”: где проверить, что считать done, что написать пользователям.

Встроенные заготовки см. в разделе “Чеклисты и release notes” ниже.

## Примеры end-to-end

Ниже 2 примера с одинаковой структурой:
- входной `task.md`
- промпты по шагам
- ожидаемые результаты (скелеты)

Примечание: примеры намеренно краткие. Если нужно — вынесем их в `examples/` как файлы для копирования.

### Пример A — Integration: подключение по API key + import/sync

#### Вход (`task.md`, фрагмент)

```markdown
Нужно добавить интеграцию с внешним сервисом: подключение по API key, импорт сущностей.
Требование: повторная синхронизация — только новых.
Нужны понятные ошибки при отзыве ключа.
```

#### Промпт 1 (routing-only)
```text
Сделай только секцию ## Routing (TaskType/MustHaveInputs/RecommendedPrompts/StopConditions/RoutingDecision).
Не генерируй Spec/AC/TestPack.
```

#### Ожидаемый routing (фрагмент)

```text
TaskType: Integration
MustHaveInputs:
- Что значит “только новые” (по какому ключу/времени/статусу)
- Ограничения API (лимиты/ошибки) если известны
RecommendedPrompts:
- 0 Intake (опционально)
- 1 Extract facts
- 2 Find gaps
- 3.1 Questions-to-author
StopConditions:
- Нет определения “new” → нельзя фиксировать AC/test
```

#### Промпт 2 (пробелы/противоречия)
```text
Сделай анализ пробелов и противоречий входной задачи.
Выведи таблицу пробелов и таблицу противоречий.
Запрет: не предлагай решений.
```

#### Промпт 3 (questions-only)
```text
Сделай Routing, затем верни только вопросы/Unknowns (P0/P1/P2) и "What to provide next".
Запрет: без Spec/AC/TestPack.
```

Ожидаемый результат (пример фрагмента P0):

```text
## P0 questions (blockers)
- Q-1: Что значит “только новые”: по timestamp? по ID? по статусу? какие исключения?
- Q-2: Что делать при revoked/expired API key: какой текст + CTA? как восстановление?
## What to provide next
- Пример ответа API / список ошибок и лимитов (если есть)
- Скрин/описание текущего UI статусов (если есть UI)
```

#### Промпт 4 (full pack)
```text
Собери полный “output pack” на основе task.v2.md и входного текста.
После сборки прогони валидаторы 9.1–9.3.
```

Ожидаемый результат (пример “что отдаём dev”):

```markdown
## Spec v1
## Acceptance Criteria
## Definition of Done
## Unknowns / Assumptions (если остались)
```

### Пример B — UX: изменить флоу/контент/статусы

#### Вход (`task.md`, фрагмент)

```markdown
Нужно упростить подключение интеграции: сейчас пользователи “не понимают, что делать дальше”.
Нужно изменить тексты и статусы на экране подключения.
```

#### Промпт 1 (routing-only)
```text
Сделай только секцию ## Routing (TaskType/MustHaveInputs/RecommendedPrompts/StopConditions/RoutingDecision).
Не генерируй Spec/AC/TestPack.
```

#### Ожидаемый routing (фрагмент)

```text
TaskType: UX
MustHaveInputs:
- Baseline: текущий флоу (шаги) + где именно “непонятно” (скрин/описание)
- Target: что считаем улучшением (наблюдаемый критерий)
RecommendedPrompts:
- 8.4 UX intake
- 1 Extract facts
- 2 Find gaps
- 3.1 Questions-to-author
StopConditions:
- Нет baseline/target → вернуть только Unknowns/P0
```

#### Промпт 2 (UX intake + пробелы)

```text
Сделай UX intake:
- Baseline (как сейчас)
- Target (что хотим)
- Inventory (что меняем: экраны/объекты/тексты)
- Constraints (если есть)
- Unknowns (P0 сверху)

Затем найди пробелы во входных данных (что мешает согласовать тексты/экраны/статусы/ошибки).
Запрет: не предлагай дизайн-решения. Если данных нет — вернуть только Unknowns/P0.
```

Ожидаемый результат (что отдаём дизайнеру, фрагмент):

```text
## Baseline (как сейчас)
- ...
## Target (что хотим)
- ...
## Inventory (что меняем)
- Экраны: ...
- Тексты: ...
## Unknowns (P0 сверху)
- ...
```

#### Промпт 3 (questions-only)

```text
Сделай Routing, затем верни только вопросы/Unknowns (P0/P1/P2) и "What to provide next".
Запрет: без Spec/AC/TestPack.
```

#### Промпт 4 (full pack)

```text
После появления baseline/target и согласованных текстов собери полный “output pack”.
После сборки прогони валидаторы 9.1–9.3.
```

## Чеклисты и release notes (копировать)

Для оценки качества результата используй универсальный чек‑лист качества (факты≠допущения, есть вопросы, строгий формат, проверяемость, нет расплывчатости, покрыты corner cases).

### Минимальный DoD “пакета” (быстро проверить себя)

```markdown
## Минимальный DoD пакета
- [ ] Есть Spec v1 со Scope/терминами/REQ/ошибками/NFR (или NFR-вопросами)
- [ ] Есть AC + DoD
- [ ] Есть тест‑матрица + тест‑план (хотя бы черновик)
- [ ] Есть список вопросов P0/P1/P2 + (если нужно) 1–3 ASSUMPTION с рисками
- [ ] Валидаторы 9.1–9.3 не находят критичных проблем, либо проблемы вынесены в Unknowns
```

### Acceptance checklist (коротко, для релиза)

```markdown
## Acceptance checklist

### Функциональность
- [ ] Все AC выполнены и проверены (ссылки на AC-1..AC-N)
- [ ] Ошибки/статусы соответствуют спеке (сообщение + CTA, без “что-то пошло не так”)

### UX/контент (если применимо)
- [ ] Тексты согласованы (RU/EN/… если есть)
- [ ] Edge cases покрыты (пустые состояния, ретраи, частичные ошибки)

### QA
- [ ] Есть тест-матрица и тест-план
- [ ] Пройдены pos/neg/corner кейсы

### Наблюдаемость/диагностика (если применимо)
- [ ] Понятно где смотреть логи/метрики для диагностики

### Документация и коммуникации
- [ ] User guide / internal notes обновлены (если нужно)
- [ ] Release notes подготовлены
```

### Release notes template (user-facing + internal)

```markdown
## Release notes (user-facing)
### Что изменилось
- ...

### Кому полезно
- ...

### Как пользоваться
1) ...

### Ограничения/известные нюансы
- ...

---

## Release notes (internal)
### Scope
- ...

### Rollout / flags (если применимо)
- ...

### Monitoring
- Метрики/алёрты:
- Где смотреть логи:

### Rollback plan (если применимо)
- Триггеры:
- Шаги:
```
