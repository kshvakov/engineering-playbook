# Реальные кейсы task-formalizer (копилка)

Цель этого файла:
- хранить **реальные кейсы** (вход → прогон gates → результаты/наблюдения);
- хранить **переиспользуемые промпты** (в т.ч. “analysis-first”);
- позже — на основе кейсов обновлять/расширять `PLAYBOOK.md` (без домыслов).

Правила:
- В кейсах фиксируем **факты и вопросы**, а не “идеальную спецификацию”.
- Важные утверждения всегда с основанием: цитата из входа или ссылка на код (файл+строки).
- Если данных нет — пишем `Unknowns` или `ASSUMPTION` (с риском и как валидировать).

## Reusable prompts

### Prompt: analysis-first (читает файл задачи сам)

```text
Твоя задача — максимально строго проанализировать ТЗ из файла и исходный код репозитория(ев). Сначала собираем факты, пробелы и противоречия. Никакой реализации/дизайна/спеки на этом шаге.

## Вход
- TaskFile: <PATH_TO_TASK_FILE>   (по умолчанию: task.md)
- OptionalFiles (если есть): <PATH_1>, <PATH_2> ...
- RelatedRepos (если есть): <REL_REPO_1_PATH>, <REL_REPO_2_PATH> ...

## Обязательные действия (в таком порядке)
1) Прочитай TaskFile и OptionalFiles. Если файла нет/пустой — остановись и верни только секцию Unknowns с просьбой предоставить файл/путь.
2) Извлеки ключевые термины/названия сущностей/кнопок/статусов/ошибок/инвариантов из ТЗ.
3) По этим терминам найди в коде релевантные места (поиск по именам сущностей, эндпоинтов, доменных объектов, UI-строк, кодов ошибок).
4) Прогони gates: Gate 0 → Gate 1 → Gate 2 → Gate 3.2 → Gate 3.1.
5) Остановись. НЕ переходи к Spec/AC/TestPack/Docs.

## Жёсткие правила
- Используй только факты из входных файлов и кода.
- Каждое важное утверждение должно иметь основание:
  - ТЗ: ссылка на раздел/цитату (и строку, если доступно),
  - Код: путь к файлу + строки (или функция/тип + строки).
  Если основания нет — это Unknowns или ASSUMPTION (с риском).
- Не “достраивай как обычно”: контракты API, схемы БД, форматы ошибок, статусы, ретраи, джобы — только если это явно есть во входе/коде.
- Не предлагай решений. В этом прогоне ты только фиксируешь проблемы и вопросы.
```

### Prompt: routing-only (только маршрутизация)

```text
Прочитай TaskFile: <PATH_TO_TASK_FILE> (по умолчанию task.md).
Сделай только секцию ## Routing.

Формат секции:
- TaskType
- MustHaveInputs
- RecommendedPrompts
- StopConditions
- RoutingDecision

Запрет:
- не генерируй Spec/AC/TestPack (даже если кажется, что всё понятно)
- не додумывай факты
```

### Prompt: questions-only (вопросы + “что предоставить”)

```text
Прочитай TaskFile: <PATH_TO_TASK_FILE> (по умолчанию task.md).
Сделай Routing, затем верни только вопросы/Unknowns (P0/P1/P2) и секцию "What to provide next".
Запрет: не генерируй Spec/AC/TestPack/тесты/дизайн.
Каждый вопрос должен приводить к конкретному решению/данным.
```

```markdown
### Prompts used (по порядку)
1) <prompt name / текст или ссылка>
2) ...

### Results (highlights)
#### Facts (самое важное, с основаниями)
- ...

#### Gaps (топ-5)
- ...

#### Contradictions (если были)
- ...

#### Questions P0 (blockers)
- ...

### Validators notes
- 9.1: где чуть не появились домыслы / как поправили
- 9.2: что было непроверяемым
- 9.3: какие термины были двусмысленны

### Postmortem
- Что в ТЗ оказалось самым “ломающим” (почему)
- Что нужно было попросить у автора заранее
- Какие guardrails в промпте помогли / не помогли
```

Дополнительный промпт: “документ для заказчика на корректировку ТЗ”

Давай из этой простыни сформируем документ, который можно отправить заказчику, чтобы он внёс коррективы в задачу.

Вход:
- Ниже в переписке есть полный текст нашей проработки/анализа (facts/gaps/contradictions/questions). Используй его как единственный источник.
Правила:
1) Не добавляй новых фактов, решений и “как лучше сделать”. Только то, что уже есть в анализе.
2) Пиши простым языком, без внутренней терминологии команды. Без ссылок на код, репозитории, названия файлов и внутренние реализации.
3) Любой спорный момент оформляй как вопрос на подтверждение/выбор.
4) Если в анализе есть противоречия — подсвети их и попроси выбрать один вариант.
5) Цель документа — чтобы заказчик мог:
   - подтвердить понимание,
   - ответить на вопросы,
   - поправить исходное ТЗ.

Формат результата (строго):

# Документ на уточнение и корректировку задачи

## 1) Как мы поняли задачу (кратко)
- 5–10 буллетов: что должно быть сделано (без деталей реализации).

## 2) Что точно входит / не входит в задачу
### Входит
- ...
### Не входит
- ...

## 3) Вопросы, без которых нельзя продолжать (P0)
- Q1: ...
  - Варианты ответа (если были в анализе): ...
  - Почему это важно: ...
- ...

## 4) Вопросы, которые можно решить позже (P1/P2)
### P1
- ...
### P2
- ...

## 5) Найденные противоречия и что нужно выбрать
- C1: <в чём конфликт>
  - Вариант A: ...
  - Вариант B: ...
  - Просьба: выберите A или B (или предложите третий).

## 6) Что нужно прислать/предоставить (артефакты)
- ...

## 7) Предлагаемые правки в текст ТЗ (черновик для копипаста)
Верни блок “заменить/добавить в задачу” в виде готовых секций:
- Контекст
- Цель
- Scope / Out of scope
- Сценарии
- Требования
- Статусы/ошибки (если в анализе были только вопросы — так и напиши, без выдумывания)

Тон:
- нейтральный, деловой, без давления, без “мы рекомендуем”.
- максимум конкретики, минимум воды.
