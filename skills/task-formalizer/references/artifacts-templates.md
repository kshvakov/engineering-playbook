# Шаблоны артефактов (копируй‑вставляй)

Эти шаблоны нужны, чтобы:
- фиксировать результат формализации в понятном и проверяемом виде;
- быстро собирать “пакет” для разработки/QA;
- стандартизировать выход (чтобы он был пригоден к вставке в таск/доки).

Рекомендация: храните “пакет” одной простынёй (один документ/один ответ), секциями ниже.

---

## 2) Acceptance Criteria (Given/When/Then)

```markdown
## Acceptance Criteria

- AC-1: Given <предусловия> When <действие> Then <ожидаемый результат>
  - Проверка: <UI/лог/метрика/данные>
  - Примечания/исключения:

- AC-2: ...
```

Подсказка по качеству AC:
- AC должны быть **наблюдаемыми** (можно проверить).
- Избегайте “понятная ошибка/валидный ключ/быстро” без определения:
  - “Понятная” → какой текст, где показан, есть ли CTA.
  - “Валидный” → какие проверки (scope/response/error codes).

## 3) Definition of Done (DoD)

```markdown
## Definition of Done

### Функциональность
- [ ] Реализованы все AC (AC-1..AC-N)
- [ ] Идемпотентность/уникальность ключей соблюдается

### Тестирование
- [ ] Есть тест-план/матрица кейсов
- [ ] Пройдены pos/neg/corner кейсы
- [ ] Проверены повторы/ретраи/частичные падения (если применимо)

### UX
- [ ] Статусы и ошибки отображаются в UI по спеке
- [ ] Сообщения об ошибках содержат понятное действие (что сделать пользователю)

### Наблюдаемость (если применимо)
- [ ] Логи позволяют диагностировать ошибки (ID интеграции, video_id, причина)
- [ ] Метрики/алёрты определены (по договорённости)

### Документация
- [ ] User guide обновлён (если нужно)
- [ ] Internal notes обновлены (если нужно)

### Релиз
- [ ] Фича-флаг/роллаут стратегия согласована (если применимо)
```
