# Core gates (0–8) — промпты для сборки пакета

Этот файл содержит **базовые** гейты (Gate 0–6b + Docs/Gate 7) для сборки полного пакета.

## Оглавление

- Общий пролог (для большинства промптов)
- 0) Intake (Gate 0)
- 1) Extract facts (Gate 1)
- 2) Find gaps (Gate 2)
- 3) Questions (Gate 3)
- 3.1) Questions-to-author (Gate 3.1)
- 3.2) Contradiction detector (Gate 3.2)
- 4) Spec v1 (Gate 4)
- 5) Acceptance Criteria + DoD (Gate 5)
- 6) Corner cases (Gate 6a)
- 7) Test pack (Gate 6b)
- 8) Docs (Gate 7)

---

## Общий пролог (добавляйте к большинству промптов)

```text
Правила:
1) Используй только факты из входных данных.
2) Если информации не хватает — не выдумывай; задай вопросы в секции Unknowns/Questions.
3) Любые допущения помечай как ASSUMPTION и объясняй, почему оно нужно.
4) Пиши кратко, инженерно, без воды.
5) Если ты утверждаешь что-то важное (правило, инвариант, формат ошибки, критерий приёмки) — добавь основание во входе (цитата/раздел). Если основания нет — перенеси в Unknowns или ASSUMPTION.
6) Не “достраивай” архитектуру/БД/эндпоинты “как обычно”. Если это нужно для артефакта — сформулируй вопрос.
7) Если не можешь ответить строго в заданном формате — верни только секцию Unknowns (и/или Questions) без остального.
```

## 0) Intake: “приведи задачу в порядок” (Gate 0)

```text
РОЛЬ: Ты системный аналитик.
ЦЕЛЬ: Преврати сырой текст задачи в:
1) читаемое резюме задачи (user-story, строго по фактам),
2) структурированный конспект (цель, scope, термины, сценарии, требования, статусы/ошибки).

ВХОДНЫЕ ДАННЫЕ:
<вставь задачу>

ОГРАНИЧЕНИЯ:
<общий пролог>

ФОРМАТ ВЫВОДА (строго):
## Task summary (rewritten)
### Проблема / контекст
...
### Цель
...
### Что меняем (scope)
...
### Критерии успеха
...
### Требуется уточнить
- ...

## Цель
## Scope / Out of scope
## Термины (таблица)
## Сценарии (нумерованный список)
## Требования (REQ-1..)
## Статусы/Ошибки
## Unknowns

КРИТЕРИИ КАЧЕСТВА:
- Никаких новых фактов.
- `Task summary (rewritten)` не должен содержать ничего, чего нет во входе; если чего-то не хватает — это уходит в `## Требуется уточнить` / `## Unknowns`.
- Unknowns содержит только конкретные неизвестные.
```

## 3.1) Questions-to-author: “сообщение автору задачи” (Gate 3.1)

```text
РОЛЬ: Ты ведущий аналитик + QA-лид.

ЦЕЛЬ: По входной задаче составь список уточняющих вопросов автору задачи, чтобы:
1) разработчик мог реализовать без домыслов,
2) QA мог протестировать,
3) UX/ошибки были однозначны.

ВХОДНЫЕ ДАННЫЕ:
<вставь текст задачи>

ОГРАНИЧЕНИЯ:
- Используй только факты из входных данных. Не добавляй “как обычно делают”.
- Если замечаешь двусмысленность — сформулируй вопрос.
- Не предлагай решения, пока не задал вопросы.
- Любые допущения оформляй отдельным блоком ASSUMPTION (с риском и способом валидации).
- Для каждого вопроса добавляй ссылку на фрагмент входа (раздел/цитата).

ФОРМАТ ВЫВОДА (строго):
## P0 (блокеры)
- Q-1 (кому): <вопрос>
  - Почему важно:
  - Что сломается без ответа:
  - Основание во входе:
## P1
...
## P2
...
## ASSUMPTION (если нужно двигаться без ответов)
- ASSUMPTION-1: ... (риск) (как валидировать)
```
