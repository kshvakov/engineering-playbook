# Рубрика качества и чек‑листы самопроверки

Цель: оценивать артефакты **как инженерные результаты**, а не как “красивый текст”.

---

## Рубрика оценки (0–2)

Оценивайте каждый артефакт по шкале 0–2:
- **2 — отлично**: проверяемо, однозначно, привязано к входу, есть неизвестные/вопросы, формат пригоден к вставке в рабочие инструменты.
- **1 — средне**: полезно, но есть двусмысленности/вода/не хватает критериев проверки/слабая привязка к входу.
- **0 — плохо**: похоже на эссе; есть домыслы; нельзя применить в разработке/QA.

---

## Универсальный чек‑лист качества (перед тем как считать “готово”)

- [ ] **Факты ≠ допущения**: допущения помечены `ASSUMPTION`.
- [ ] **Есть вопросы** там, где вход молчит (нет выдуманных деталей).
- [ ] **Формат строгий** (таблица/чек‑лист/нумерация), удобно копировать.
- [ ] **Проверяемость**: можно объяснить, как QA/разработчик проверит каждый пункт.
- [ ] **Нет расплывчатости** (“понятный”, “валидный”, “быстро”) без определения.
- [ ] **Есть coverage corner cases** (ретраи, частичные ошибки, повторы, конкуренция).

---

## Быстрые критерии DoD для “пакета”

Пакет считается “минимально достаточным”, если:
- [ ] Есть Spec v1 (PRD‑lite) со Scope/терминами/REQ/ошибками/NFR‑вопросами.
- [ ] Есть AC + DoD.
- [ ] Есть тест‑матрица + тест‑план (хотя бы черновик).
- [ ] Есть список вопросов P0/P1/P2 + (если нужно) 1–3 `ASSUMPTION` с рисками.
- [ ] Валидаторы (галлюцинации/проверяемость/двусмысленность) не находят критичных проблем, либо проблемы вынесены в Unknowns.

---

## Мини‑паттерны “чинить качество”

Если результат “плывёт”:
- Ужесточить формат: “Если не можешь ответить строго — верни только Unknowns”.
- Ограничить объём: “не более 10 пунктов, отсортируй по риску (P0 сверху)”.
- Перевернуть порядок: “сначала Questions/Unknowns, потом Draft”.

Если модель начинает “советовать как обычно”:
- Добавить запрет: “не достраивай архитектуру/эндпоинты/таблицы”.
- Требовать основание: “каждое важное утверждение должно иметь цитату/раздел”.
